# Журнал действий пользователей

## Цель

Сделать в админке понятный журнал действий: кто выполнил действие, над каким объектом, в каком модуле, когда и с каким результатом.

## Пользовательская ценность

- Администратор быстро разбирает спорные ситуации без чтения технических логов.
- Руководитель видит историю изменений по пользователям, ролям, проектам, сметам, платежам, складу и закупкам.
- Поддержка может ответить на вопрос "что произошло" по человеку, проекту или объекту.
- Система фиксирует важные действия в едином формате и не показывает пользователю внутренние технические поля.

## Не является целью

- Не заменяет GlitchTip/Sentry и технические Laravel-логи.
- Не показывает пользователю сырой JSON, SQL, stack trace, payload или DTO.
- Не логирует успешный фоновый polling, открытие списков и обычные dashboard-refresh действия.
- Не хранит пароли, токены, ключи, authorization headers и необработанные тела запросов.

## Основные сценарии

1. Администратор открывает раздел "Журнал действий" и видит последние события организации.
2. Администратор фильтрует события по пользователю, модулю, проекту, объекту, результату и периоду.
3. Администратор открывает событие и видит понятное описание, связанный объект и безопасный список изменений.
4. Администратор экспортирует отфильтрованный журнал в CSV.
5. Поддержка проверяет заблокированные действия, ошибки доступа и подозрительные события.

## Модель события

Каждое событие хранится в таблице `activity_events` и содержит:

- организацию;
- пользователя-исполнителя или системного исполнителя;
- интерфейс, из которого выполнено действие;
- модуль;
- тип события;
- действие;
- результат;
- важность;
- объект действия;
- проект, если действие связано с проектом;
- целевого пользователя, если действие направлено на пользователя;
- человекочитаемые заголовок и описание;
- безопасный список изменений;
- безопасный контекст;
- IP, user agent и correlation id;
- дату события.

## Права

- `activity-events.view` — просмотр журнала.
- `activity-events.export` — экспорт журнала.
- `activity-events.view_security` — просмотр событий безопасности.
- `activity-events.view_technical_context` — просмотр технического контекста в свернутом виде.

Обычные проектные роли не получают глобальный журнал действий по умолчанию. Системный администратор получает все права. Поддержка получает просмотр без экспорта и без технического контекста, если роль не расширена явно.

## API

Backend должен отдавать данные через `AdminResponse`:

- `GET /api/v1/admin/activity/events` — список событий с пагинацией.
- `GET /api/v1/admin/activity/events/{id}` — детальная карточка события.
- `GET /api/v1/admin/activity/summary` — агрегаты для верхней панели.
- `GET /api/v1/admin/activity/actors` — список пользователей для фильтра.
- `GET /api/v1/admin/activity/events/export` — экспорт CSV.

Фильтры:

- пользователь;
- целевой пользователь;
- проект;
- модуль;
- тип события;
- действие;
- результат;
- важность;
- тип и ID объекта;
- дата с/по;
- поисковая строка по безопасным пользовательским полям.

## UX админки

Первый экран — рабочий реестр, а не техническая таблица.

Обязательные зоны:

- заголовок "Журнал действий";
- краткая сводка по количеству событий, ошибкам, предупреждениям и активным пользователям;
- фильтры;
- таблица или лента событий;
- detail drawer с безопасными подробностями;
- экспорт.

В таблице показываются:

- дата и время;
- пользователь;
- действие;
- модуль;
- объект;
- проект;
- результат.

В detail drawer показываются:

- описание события;
- исполнитель;
- объект;
- проект;
- список измененных полей;
- безопасные детали;
- технический контекст только при отдельном праве.

## Безопасность и редактирование данных

Перед записью в `activity_events` удаляются или заменяются на `[скрыто]`:

- пароли;
- токены;
- секреты;
- API-ключи;
- authorization headers;
- сырые request body;
- персональные данные, не нужные для бизнес-разбора.

Событие не должно ломать бизнес-операцию. Если запись журнала не удалась, система пишет техническую ошибку и продолжает основное действие.

## Первый production-срез

В первую очередь журналируются:

- вход и неуспешный вход;
- создание и изменение пользователей;
- назначение и отзыв ролей;
- создание и изменение проектов;
- ключевые события закупок;
- создание, согласование и отклонение платежных документов;
- импорт и изменение смет;
- складские операции с активами и движениями.

## Rollout

1. Добавить миграцию и backend-ядро.
2. Добавить API и права.
3. Добавить admin UI.
4. Подключить первый набор событий.
5. Проверить на staging.
6. Во время релиза выполнить миграции вручную.
7. Включить запись событий.
8. Проверить список, фильтры, detail drawer и экспорт.
9. При необходимости выполнить dry-run переноса старых audit-событий.

## Критерии готовности

- У каждой записи есть исполнитель, действие, объект или контекст, результат и дата.
- Админка показывает понятный русский текст, а не внутренние ключи.
- Фильтры и пагинация работают на больших объемах.
- Экспорт уважает фильтры и права.
- Данные ограничены текущей организацией.
- Чувствительные данные не сохраняются в журнал.
- Старый экран системных логов больше не является основным пользовательским сценарием.
- Backend и frontend покрыты тестами.
- Документация в репозитории и Notion обновлена.
