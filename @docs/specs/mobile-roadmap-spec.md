# Спецификация: Roadmap мобильного приложения ProHelper

## Цель
Сформировать единый план развития `prohelpers_mobile`, в котором первоочередной задачей является исправление битых текстов и переводов в мобильном складе, а далее последовательное доведение приложения до стабильного продуктового состояния для полевого использования.

## Контекст
- Мобильное приложение уже покрывает базовые сценарии: авторизация, дашборд, проекты, заявки с объекта, склад, графики работ.
- Основные пользовательские боли сейчас сосредоточены в качестве mobile UX, целостности переводов, устойчивости API-контрактов и отсутствии производственного контура выпуска.
- Приоритет №1 на текущем этапе: модуль склада, включая тексты, переводы, устойчивость отображения и базовую пригодность для ежедневной работы.

## Проблемы текущего состояния

### P0. Критические
- В мобильном складе встречаются битые строки и ключи переводов вместо пользовательских текстов.
- Часть mobile-переводов хранится в битой кодировке на backend.
- Мобильный клиент недостаточно защищен от сырых translation keys в ответах API.
- Нет минимального тестового контура на критические пользовательские сценарии.

### P1. Высокий приоритет
- Нет нормального разделения окружений `dev/stage/prod`.
- Release-сборка не оформлена как полноценный production-процесс.
- Дашборд и быстрые действия частично зависят от нестабильных текстовых ответов backend.
- Основные модули не доведены до полноценного мобильного UX уровня.

### P2. Средний приоритет
- Нет crash-reporting и аналитики мобильного приложения.
- Нет offline/light-offline поведения.
- Нет push-уведомлений по ключевым событиям.
- Не оформлен единый QA-чеклист и сценарии регрессии.

## Продуктовые принципы
- Любой текст, уходящий пользователю, должен быть стабильно читаемым даже при деградации перевода на сервере.
- Мобильное приложение должно быть полезно для прораба и полевого сотрудника без обращения к админке для ежедневных действий.
- Приоритет отдается сценариям “посмотреть текущее состояние”, “быстро понять проблему”, “сразу выполнить действие”.
- Каждый критичный экран должен иметь состояния `loading`, `error`, `empty`, `content`.

## Roadmap

### Этап 1. Стабилизация и доверие к интерфейсу

#### 1.1. Склад и переводы
- Исправить backend-переводы `mobile_warehouse` и связанные строки `mobile_dashboard`.
- Добавить на клиенте fallback-мэппинг для складских заголовков, описаний и типов движений.
- Проверить все тексты склада: заголовки, пустые состояния, ошибки, кнопки, статусы, валюту.
- Убедиться, что склад остается читаемым при пустых/битых значениях `movement_type_label`.

#### 1.2. Общая текстовая консистентность
- Пройти dashboard, быстрые действия и ключевые карточки модулей на предмет translation keys.
- Устранить mojibake в client-side fallback-строках.
- Зафиксировать правило: мобильный клиент не должен напрямую показывать ключи `mobile_*`.

#### 1.3. Базовый контроль качества
- Добавить минимум smoke/widget тесты:
  - вход в приложение,
  - выбор объекта,
  - открытие склада,
  - открытие графиков работ,
  - открытие заявок.
- Добавить unit-тесты на парсинг DTO, где используются fallback-тексты.

### Этап 2. Production-ready контур

#### 2.1. Сборка и окружения
- Вынести конфигурацию `baseUrl` в окружения `dev/stage/prod`.
- Настроить понятный release flow для Android:
  - signing,
  - versioning,
  - инструкция сборки,
  - инструкция установки на тестовые устройства.
- Зафиксировать поддерживаемый Android SDK и проверку окружения.

#### 2.2. CI и проверка регрессий
- Добавить автоматический запуск:
  - `flutter analyze`,
  - unit tests,
  - widget tests.
- Подготовить чеклист ручной регрессии для тестовой APK-сборки.

### Этап 3. Доведение ключевых модулей до продуктового уровня

#### 3.1. Склад
- Улучшить карточки складов:
  - тип склада,
  - адрес,
  - ценность остатков,
  - признаки “основной”, “дефицит”.
- Добавить поиск/фильтрацию по материалам и движениям.
- Добавить отдельный акцент на дефицит и резервы.
- Подготовить сценарии “что закончилось”, “что недавно пришло”, “что зарезервировано”.

#### 3.1.1. Зафиксированное состояние warehouse UX/admin на текущем этапе
- Backend склада уже содержит сильную базу для enterprise-сценария: остатки, движения, резервы, инвентаризацию, зоны хранения, партии, серийники, сроки и адресные поля `location_code`, `rack_number`, `shelf_number`, `cell_number`.
- Текущий admin UI склада перегружен верхнеуровневой навигацией: на одном экране одновременно живут обзор, остатки, документные операции, инвентаризация, резервы, аналитика, автопополнение и зоны.
- Текущий mobile UI склада является обзорным, а не операционным: пользователь видит склады, последние движения и может оформить приход, но не получает scan-first сценарий, маршрутизацию по типу кода и быстрый переход в складскую операцию после идентификации объекта.
- В текущем контуре нет самостоятельного UI-слоя для складских идентификаторов, scan resolver, логистических единиц, задач кладовщика и работы с ячейками как с основным объектом операционного процесса.

#### 3.1.2. UX-проблемы, которые нельзя усиливать при дальнейшем развитии
- Нельзя продолжать рост склада через добавление новых равноправных вкладок в admin. При добавлении ячеек, кодов, паллет, контейнеров, инструментов и RFID текущая схема станет неоперабельной.
- Нельзя оставлять ключевые складские действия скрытыми в контекстных меню таблицы остатков. Для операционного склада резерв, перемещение, выдача, пересчет, история и действия по месту хранения должны быть видимы без глубокого поиска.
- Нельзя держать адресное хранение только как строковое поле `location_code` и отдельный CRUD зон. Для будущего контура нужен полноценный браузер топологии хранения.
- Нельзя строить mobile UX вокруг просмотра списков и истории, если целевая модель склада предполагает работу “подошел к ячейке/товару/палете -> отсканировал -> увидел контекст -> выполнил действие”.

#### 3.1.3. Целевая информационная архитектура склада
- Склад в admin должен быть перестроен в операционный shell с 4 верхнеуровневыми разделами:
  - `Обзор`: KPI, критические остатки, активные инвентаризации, проблемные ячейки, ожидающие действия.
  - `Операции`: приемка, размещение, перемещение, выдача на объект, возврат, списание, инвентаризация, резервы.
  - `Хранение`: топология склада `склад -> зона -> ряд/стеллаж -> полка -> ячейка`, загрузка мест, содержимое, история, печать меток.
  - `Остатки и активы`: поиск по SKU, коду, партии, серийнику, ячейке, объекту хранения и быстрый переход в карточку.
- Аналитика, автопополнение и системные настройки склада не должны конкурировать с операционными разделами на первом уровне. Их следует вынести во вторичный раздел.
- Все будущие сущности `warehouse_identifiers`, `warehouse_storage_cells`, `warehouse_logistic_units`, `warehouse_tasks`, `scan_events`, `rfid_sessions` должны встраиваться именно в эту структуру, а не появляться как новые отдельные вкладки.

#### 3.1.4. Целевая mobile-модель склада
- Главным действием mobile-экрана склада должен стать `Сканировать`, а не просмотр списка складов.
- После сканирования backend должен через scan resolver определять тип объекта: ячейка, товар, паллета, короб, контейнер, документ, инструмент, RFID-единица.
- Экран после скана должен открывать не просто карточку, а следующий допустимый сценарий: переместить, принять, разместить, пересчитать, выдать, вернуть, списать, посмотреть историю.
- Mobile должен получить отдельные сценарии для:
  - операций по заданиям,
  - работы по ячейкам,
  - выдачи и возврата инструмента/оборудования,
  - офлайн-очереди операций для стройплощадки.

#### 3.1.5. Прямое влияние backend-спеки на UX
- Так как на backend уже существуют операции приема, списания, перемещения, резервирования и инвентаризации, UI должен перестать дублировать документную модель “по вкладкам” и перейти к сценарной модели “по действиям”.
- Так как в модели уже есть поля партий, серийников, сроков и локаций, карточки остатков и акты инвентаризации должны проектироваться как многослойные сущности, а не как плоские строки таблиц.
- Так как зоны уже имеют поля адресации и вместимости, следующий UI-этап обязан показывать загрузку и структуру хранения, а не только справочник зон.
- Так как отдельный слой идентификаторов и scan-flow еще не реализован, все новые UI-изменения должны учитывать будущую интеграцию scan resolver и не завязываться на ручной ввод как на основную модель работы.

#### 3.1.6. Зафиксированная foundation-реализация, на которую опирается следующий этап
- На backend должны существовать отдельные сущности `warehouse_storage_cells` и `warehouse_identifiers`, а не только строки `location_code` и CRUD зон.
- Для admin должен быть доступен полноценный storage-раздел из трех рабочих экранов: `зоны`, `ячейки`, `идентификаторы`, где `ячейки` и `идентификаторы` являются не декоративными вкладками, а самостоятельными сценариями управления.
- Контур `advanced-warehouse` обязан быть выровнен с admin-клиентом по контрактам резервов и автопополнения: реальные маршруты, реальные поля ответов и update/delete-операции для правил.
- Scan-ready слой начинается уже в admin: код должен уметь резолвиться в сущность склада до появления мобильного сканера, чтобы UI и backend не расходились по модели идентификации.
- В `каталоге активов` должен существовать печатный экспорт QR-листов в PDF с сеткой 6 или 4 меток на страницу, где каждая метка содержит QR, название, артикул и тип позиции и пригодна для физической печати и нарезки на складе.

#### 3.2. Графики работ
- Доработать мобильный список графиков до уровня админки:
  - фильтры,
  - поиск,
  - понятные статусы,
  - акценты на просрочку и риски.
- Улучшить экран деталей графика:
  - прогресс,
  - просроченные задачи,
  - критический путь,
  - группировка задач.

#### 3.3. Заявки с объекта
- Довести сценарии создания и обработки заявок:
  - вложения,
  - черновики,
  - история статусов,
  - повторная отправка после ошибки,
  - понятные состояния согласования.

#### 3.4. Dashboard
- Превратить dashboard в стартовую точку рабочего дня:
  - новые заявки,
  - просроченные задачи,
  - дефицит материалов,
  - ожидающие согласования.
- Сократить декоративные карточки в пользу actionable-summary.

### Этап 4. Зрелость продукта

#### 4.1. Наблюдаемость
- Подключить crash-reporting.
- Добавить аналитику ключевых пользовательских действий.
- Фиксировать ошибки API по модулям и экранам.

#### 4.2. Offline и устойчивость
- Кешировать последний выбранный объект.
- Кешировать последние данные по складу, заявкам и графикам.
- Сделать graceful fallback при потере сети.

#### 4.3. Уведомления
- Добавить push-уведомления:
  - по новым заявкам,
  - по согласованиям,
  - по критичным задачам,
  - по складским дефицитам.

## Зафиксированное warehouse foundation-расширение
- После внедрения `warehouse_storage_cells` и `warehouse_identifiers` складской контур должен включать `warehouse_logistic_units` и `warehouse_scan_events` как обязательный слой для enterprise-модели.
- `warehouse_logistic_units` покрывают паллеты, короба, контейнеры, связки и другую групповую тару, которую можно адресно размещать в зонах и ячейках, вкладывать друг в друга и маркировать отдельными QR/RFID-метками.
- `warehouse_scan_events` формируют единый журнал сканирований для admin/mobile/ТСД/RFID-источников, чтобы видеть распознанные и нераспознанные коды, отлаживать resolver и готовить почву для scan-driven операций.
- В admin storage-разделе эти сущности должны быть представлены отдельными рабочими сценариями `Логистические единицы` и `Журнал сканов`, а не дополнительными декоративными вкладками.

## Зафиксированное warehouse task-расширение
- Следующий обязательный слой поверх `warehouse_storage_cells`, `warehouse_identifiers`, `warehouse_logistic_units` и `warehouse_scan_events` — это `warehouse_tasks` как операционная очередь склада.
- `warehouse_tasks` должны покрывать как минимум типы `receipt`, `placement`, `transfer`, `picking`, `cycle_count`, `issue`, `return`, `relabel`, `inspection` и хранить статусную машину `draft -> queued -> in_progress -> blocked -> completed/cancelled`.
- Задача склада обязана ссылаться на контекст хранения и исполнения: склад, зону, ячейку, логистическую единицу, материал, проект, исполнителя и при необходимости на исходный документ (`inventory_act`, `movement`, source document).
- В admin задачи должны быть выведены в секцию `Операции` как самостоятельный сценарий `Задачи`, а не растворены внутри форм приемки, списания и перемещения.
- Экран задач должен быть ориентирован на работу по очереди: фильтры, список задач, карточка выбранной задачи, явные статусные действия, привязка к адресному хранению и минимизация ручного поиска по разным табам.
- Mobile/scan-flow должен в следующем этапе опираться именно на `warehouse_tasks`: после скана объект должен открывать не просто карточку, а релевантное задание или предложенный следующий шаг выполнения.
