# Task 6: удаление старой тарифной модели

## Результат

Из runtime backend МОСТ удалена старая модель тарифных планов и подписок. Коммерческий доступ определяется новым контуром: `OrganizationCommercialAccount`, `OrganizationPackageSubscription`, `CommercialOrder` и `CommercialPayment`.

Удалены старые модели, сервисы, контроллеры, middleware, команды продления, Filament-ресурсы и уведомления. `OrganizationModuleActivation` оставлена только как техническое хранилище состояния, настроек и метрик использования без связей со старой оплатой.

Демонстрационный сценарий BrickHouse сохранён. Набор из 12 пользователей в сервисе удаления и проверки теперь точно совпадает с адресами `@most.test`, которые создаёт seeder; это закреплено отдельным контрактным тестом.

## Необратимая миграция

`2026_07_15_000001_remove_legacy_billing_runtime.php` выполняется после создания нового коммерческого контура и:

- сохраняет строки аудита реферальной программы;
- переименовывает старую ссылку `invited_subscription_id` в nullable-поле `legacy_invited_subscription_id`, не выдавая старый идентификатор за идентификатор нового заказа;
- добавляет nullable-ссылки на `commercial_orders` и `commercial_payments`, если их ещё нет;
- отменяет старые pending-награды без подтверждённого коммерческого источника с причиной `commercial_source_not_found_after_billing_migration`;
- удаляет внешние ключи и колонки старых подписок из платежей, баланса и технических активаций модулей;
- удаляет таблицы старых планов, подписок, дополнений и платежей.

Миграция намеренно необратима: `down()` всегда выбрасывает исключение и не создаёт пустые заменители удалённых таблиц. Откат возможен только восстановлением резервной копии базы перед возвратом предыдущей версии приложения. Резервная копия обязательна до production-деплоя.

## Реферальная программа

Новые награды создаются только от первого успешно оплаченного `CommercialOrder` и связанного `CommercialPayment`. Начисление выполняется после окончания оплаченного периода через bonus balance ledger.

Баланс блокируется через `FOR UPDATE`, а сервис начисления возвращает созданный `BalanceTransaction`. Награда сохраняет точные идентификаторы двух созданных проводок. Поведенческий тест подтверждает последовательное начисление двух наград одному пригласившему: общий баланс равен сумме наград, идентификаторы проводок различаются, а `meta.referral_reward_id` каждой проводки указывает на точную награду.

## Корпоративный контур

Для аккаунта со статусом `corporate` серверный guard запрещает self-service изменения: создание checkout, запуск пробного доступа и планирование изменения состава пакетов. Guard выполняется до обработки idempotency key, поэтому повтор ранее созданного запроса также возвращает единое переведённое сообщение с HTTP 409. Read-only расчёт остаётся доступен.

Если изменение состава было запланировано до перевода аккаунта в корпоративный режим, renewal на расчётной дате переводит его в `canceled`, восстанавливает затронутые строки `scheduled_for_removal` в `active` и очищает `cancel_at`. Восстановление ограничено источниками `paid_package` и `full_suite`; строки корпоративного доступа не изменяются.

Рекомендации рабочего профиля организации больше не содержат `open_modules` и `/dashboard/modules`: действие называется «Подобрать пакеты», использует ключ `open_packages` и ведёт на `/dashboard/billing`. Для профилей аренды техники и поставки материалов эта же страница используется как маршрут по умолчанию.

## Проверки

- Красная фаза TDD зафиксировала отсутствие корпоративного guard и несовпадение адресов BrickHouse.
- Полный набор из шести затронутых test-файлов проходит: 76 тестов, 350 утверждений.
- После типизации пользователя checkout-контроллера повторно проходят 3 целевых API-теста, 16 утверждений.
- PHP syntax: 16 изменённых PHP-файлов, ошибок нет.
- Laravel Pint: 16 изменённых PHP-файлов, стиль исправлен и повторно проверен.
- Larastan/PHPStan: 9 изменённых runtime-файлов с `--memory-limit=1G`, ошибок нет.
- Дополнительная проверка корпоративной отмены: красная фаза воспроизвела обход guard и зависшее scheduled-изменение; после исправления 3 целевых теста и 23 утверждения проходят.
- Полный повторный набор checkout API, renewal и schema contract: 62 теста, 341 утверждение, assertion failures отсутствуют.
- Для дополнительных исправлений Pint и PHP syntax проходят по 11 PHP-файлам, Larastan — по 5 runtime-файлам.
- Boundary-тест каталога рабочих профилей проходит: 7 тестов, 15 утверждений; в runtime-каталоге отсутствуют старые ключ, текст и маршрут модулей.

Предупреждения PHPUnit в изолированном worktree связаны только с отсутствующим `.env`; assertion failures в зелёной фазе отсутствуют.
