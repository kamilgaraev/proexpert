# Изоляция контуров уведомлений МОСТ

## Цель

Исключить попадание уведомлений между админкой, личным кабинетом, мобильным приложением и кабинетом заказчика как в реальном времени, так и после повторной загрузки данных. Одно бизнес-событие может быть явно направлено в несколько контуров, но каждый контур имеет независимые состояния прочтения и скрытия.

## Подтверждённые проблемы

1. `NotificationController` ограничивает запросы только получателем. Список, карточка, счётчики, прочтение, массовое прочтение и удаление не учитывают контур.
2. WebSocket разделён на каналы `.admin` и `.lk`, но сохранённая запись доступна через все API этого пользователя.
3. Нормализатор по умолчанию выбирает `admin`, а WebSocket — `lk`. Часть отправителей не задаёт контур явно.
4. Системные рассылки создают значение `customer`, которое текущий WebSocket-канал отклоняет.
5. Соединение Echo не всегда закрывается при выходе. Админка может оставить старую приватную подписку, а ЛК может переиспользовать соединение со старым токеном.
6. Маршруты ЛК используют общий контроллер и формат ответа админки. Интеграционных тестов изоляции для ЛК нет.
7. Список без ограничения принимает `per_page` и загружает внутреннюю аналитику доставки для каждой записи.

## Выбранный подход

Использовать отдельную таблицу целей уведомления. Содержимое и получатель остаются в `notifications`, а видимость и пользовательское состояние каждого контура хранятся в `notification_targets`.

Массив контуров внутри JSON не выбран, потому что он оставляет общие состояния прочтения и удаления. Дублирование полной записи на каждый контур не выбрано из-за риска повторной доставки email и Telegram и усложнения аналитики.

## Модель данных

### `notifications`

Таблица продолжает хранить:

- тип и содержимое;
- пользователя-получателя;
- организацию и бизнес-контекст;
- приоритет;
- внешние каналы доставки;
- обязательные права получателя в `required_permissions` для аудита решения об отправке.

Поле `read_at` временно сохраняется для обратной совместимости, но перестаёт быть источником состояния интерфейса.

### `notification_targets`

Каждая запись содержит:

- `notification_id`;
- `interface`: `admin`, `lk`, `mobile` или `customer`;
- `read_at`;
- `dismissed_at`;
- `websocket_status`, `websocket_delivered_at` и безопасное диагностическое состояние последней ошибки;
- временные метки.

Ограничения:

- уникальность пары `notification_id + interface`;
- внешний ключ с каскадным удалением;
- проверка допустимого значения `interface`;
- индексы для выборок по интерфейсу, состоянию прочтения и уведомлению.

Удаление из интерфейса означает установку `dismissed_at` только для текущей цели. Физическая запись уведомления удаляется политикой хранения после скрытия всех целей или истечения срока хранения.

## Создание и доставка

Вводится enum `NotificationInterface` и типизированный объект параметров доставки. Новые вызовы обязаны явно передавать один или несколько контуров. Пустой или неизвестный набор отклоняется до создания записи.

Переходный адаптер принимает существующее `data.interface`, пока все известные отправители не переведены на новый контракт. После миграции отправителей неявное значение по умолчанию удаляется.

Порядок отправки:

1. Нормализовать бизнес-данные без назначения интерфейса по умолчанию.
2. Проверить пользователя, организацию и обязательные права.
3. В одной транзакции создать уведомление и его цели.
4. После фиксации транзакции отправить фоновые задания.
5. Для WebSocket обработать каждую цель отдельно и сохранить успешность по цели. Повторная попытка не отправляет повторно уже доставленные цели.
6. Email и Telegram отправляются один раз на логическое уведомление, независимо от количества интерфейсов.

Для `customer` и `mobile` WebSocket разрешается только при наличии соответствующего клиента и маршрута авторизации. До этого сочетание такого контура с каналом `websocket` отклоняется явно, а не падает в очереди.

## Изоляция API

Контур определяется сервером по группе маршрутов:

- `/api/v1/admin/*` → `admin`;
- `/api/v1/landing/*` → `lk`;
- `/api/v1/mobile/*` → `mobile`;
- `/api/v1/customer/*` → `customer`.

Параметр запроса не может изменить контур. Общий сервис выборки всегда применяет одновременно:

- пользователя-получателя;
- текущую организацию, когда она указана у уведомления;
- цель текущего контура;
- `dismissed_at IS NULL`.

Одинаковая область применяется к списку, счётчикам, карточке, прочтению, массовому прочтению и скрытию. Идентификатор уведомления другого контура возвращает `404`, не раскрывая факт его существования.

Каждая группа маршрутов использует свой стандарт ответа: `AdminResponse`, `LandingResponse`, `MobileResponse` или `CustomerResponse`. Клиентские сервисы нормализуются без разрыва текущего пользовательского интерфейса.

`per_page` ограничивается диапазоном 1–100. Внутренняя аналитика доставки исключается из списка и доступна только через административный диагностический контракт с отдельным правом.

## Получатели и права

Контур отвечает за место отображения, а права — за выбор пользователей. Чувствительное уведомление создаётся только для пользователей, прошедших `AuthorizationService` в контексте организации и проекта.

Все бизнес-отправители обязаны передавать доменное право либо использовать зарегистрированное соответствие типа уведомления праву. Неизвестный чувствительный тип не должен молча превращаться в рассылку без ограничений. Решение о доступе сохраняется в `required_permissions` и журналируется без содержимого уведомления.

Доступ к записи остаётся персональным: другой пользователь не может получить её даже при совпадении роли и контура.

## Клиенты

Админка и ЛК подписываются только на свой канал и принимают только событие с совпадающим контуром. При выходе, смене пользователя или токена клиент:

1. покидает приватный канал;
2. полностью отключает Echo;
3. очищает локальные уведомления и счётчики;
4. при новом входе создаёт соединение с новым токеном.

События дедуплицируются по `notification_id + interface`, чтобы повторная доставка или повторное подключение не увеличивали счётчик дважды.

## Миграция данных

Миграция не удаляет существующие уведомления.

- Известное `data.interface` переносится в соответствующую цель.
- Отсутствующее значение переносится в `admin`, поскольку это действующее значение нормализатора.
- Неизвестные значения не публикуются автоматически; они фиксируются в диагностическом отчёте и требуют явного сопоставления.
- Текущее `read_at` копируется в созданные цели.

Миграция должна быть повторяемой и создавать цели пакетами, не загружая всю таблицу в память. Запуск миграции выполняется только штатным процессом развёртывания, не из локального окружения Codex.

## Обработка ошибок и наблюдаемость

- Ошибка одного WebSocket-контура не отменяет успешную доставку в другие контуры.
- Повторные попытки ограничены количеством и задержкой.
- Логи содержат идентификатор уведомления, контур, канал и номер попытки, но не текст и конфиденциальные данные.
- Неизвестный контур и отсутствие целей являются доменными ошибками и не создают задание очереди.
- Метрики отдельно показывают ошибки публикации по каждому контуру.

## Проверки

Backend-тесты должны доказать:

- отсутствие уведомлений ЛК в списке и счётчике админки и наоборот;
- `404` для просмотра, прочтения и скрытия уведомления другого контура;
- независимость `read_at` и `dismissed_at` целей;
- корректное массовое прочтение только текущего контура;
- публикацию в несколько WebSocket-каналов без повторов успешной цели;
- однократную отправку email/Telegram для нескольких целей;
- отклонение неизвестных и неподдерживаемых контуров;
- проверку обязательных прав и организационного контекста;
- безопасный пакетный перенос старых данных.

Клиентские тесты должны доказать:

- подписку на правильный канал;
- дедупликацию события;
- очистку состояния и отключение Echo при выходе;
- создание нового соединения при смене пользователя;
- корректную нормализацию ответов каждого API.

После развёртывания выполняется smoke-проверка отдельным тестовым уведомлением для `admin` и `lk`: событие должно моментально появиться только в целевом интерфейсе и не появиться в другом даже после обновления страницы.

## Границы задачи

В задачу входят подсистема уведомлений backend, API админки/ЛК/мобильного приложения, клиентские подписки админки и ЛК, совместимость существующих отправителей и тесты. Переработка визуального дизайна уведомлений и добавление нового WebSocket-интерфейса кабинету заказчика не входят; неподдерживаемые сочетания будут отклоняться явно.
