# Заявки и цепочка обеспечения: design исправлений

## Цель

Устранить три связанных дефекта без временных клиентских обходов:

1. карточка заказа должна показывать согласованный серверный маршрут обеспечения;
2. черновик заявки должен быть приватен для автора во всех каналах утечки;
3. ручная материальная позиция должна поддерживать закупку и не открывать пустой диалог.

## Выбранный подход

### 1. Единый snapshot цепочки заказа

`ProcurementChainService` остается источником истины для текущего этапа, блокеров и следующего действия. Детальный endpoint заказа возвращает полную `procurement_chain` в том же ответе, что и заказ. Списковые endpoint-ы сохраняют только compact summary, чтобы не увеличить payload и не создать N+1.

Admin больше не вычисляет бизнес-этапы через `buildTimelineSteps(order)`. Экран использует серверные `current_stage`, `stages`, `blockers` и `next_action`. Если нужен отдельный операционный статус заказа, он подписывается именно как статус заказа и не изображается второй сквозной цепочкой.

Существующие `workflow_summary` и `procurement_chain_summary` остаются backward-compatible. Они не удаляются и не переименовываются.

### 2. Приватность черновика как серверное правило

Для пользователя организации видимая выборка заявок определяется правилом:

```text
status != draft OR user_id = current_user_id
```

Правило применяется к списку, карточке, группам, статистике, просроченным заявкам и mobile detail. Чужой черновик возвращает `404`, чтобы API не подтверждал существование записи.

Изменяющие операции дополнительно защищаются доменными guard-ами:

- редактировать, удалить и отправить можно только собственный черновик;
- удаление заявки не в статусе draft возвращает `422`;
- draft-группа приватна автору и изменяется/отправляется только им.

Черновик не публикуется в общий календарь и не создает уведомления для других сотрудников. Публикация происходит при переходе `draft -> pending`. Роли не хардкодятся; базовое правило приватности одинаково для всех ролей. Отдельное право просмотра чужих черновиков не вводится, потому что бизнес-требование сформулировано как личный черновик.

### 3. Ручной материал поддерживается в закупке

Ручная позиция является допустимой доменной сущностью. Запрещать ее создание или согласование нельзя: закупочные линии уже допускают `material_id = null`.

GET вариантов обеспечения для ручного материала возвращает успешный контракт:

```json
{
  "material_source": "manual",
  "warehouse_lookup_supported": false,
  "warehouse_unavailable_reason": "Материал не связан со справочником. Доступна закупка, проверка склада невозможна.",
  "can_use_warehouse": false,
  "can_use_purchase": true,
  "can_use_mixed": false,
  "recommended_source": "purchase",
  "warehouses": []
}
```

Для ручного материала POST принимает только `source=purchase`. `warehouse` и `mixed` возвращают `422` до побочных эффектов. Повторный POST остается идемпотентным и не создает вторую закупочную заявку.

Admin показывает вариант закупки, имя и количество материала, а также понятное inline-предупреждение об отсутствии складской проверки. Ошибка загрузки отображается внутри диалога с повтором, поэтому пустого modal state больше нет. Глобальный API interceptor остается единственным владельцем toast для стандартных ответов; диалог не дублирует сообщения.

## Альтернативы и причины отказа

### Обязательная привязка к каталогу

Вариант отклонен: он ломает существующие записи и поддерживаемый ручной procurement, требует миграции данных и запрещает согласование заявок, которые сегодня считаются валидными.

### Только переименовать две цепочки

Вариант отклонен как косметический: локальный калькулятор уже ошибается на `partially_delivered`, а два последовательных HTTP-запроса могут вернуть разные snapshots.

### Скрыть черновики только в admin list

Вариант отклонен как небезопасный: прямой URL, группы, статистика, mobile detail, уведомления и календарь продолжат раскрывать данные.

## Ошибки и совместимость

- Незнакомые или отсутствующие новые поля трактуются клиентом безопасно.
- Ошибка полной цепочки в detail отображается явно; клиент не восстанавливает бизнес-стадию из enum.
- Старые compact-поля сохраняются.
- Manual fulfillment не разрешает складские операции без `material_id`.
- Все пользовательские backend-сообщения используют `trans_message(...)`.

## Проверка

- Backend feature-тесты покрывают приватность, группы, статистику, mobile detail, уведомления/календарь, manual fulfillment, идемпотентность и full chain detail.
- Admin Vitest покрывает нормализацию detail-контракта, server-driven stages, manual purchase-only dialog, inline error/retry и отсутствие двойных toast.
- Выполняются `php -l`, Larastan/PHPStan, `npx tsc --noEmit`, точечный Vitest и browser smoke при доступном URL.
- Миграции, локальные DB-команды, dev-server и admin build не запускаются.
