# Контур доказательной готовности AI-сметы

**Дата:** 2026-07-22
**Статус:** согласовано пользователем, готово к реализации
**Область:** генератор AI-смет МОСТ; сессия №59 используется как регрессионный сценарий

## Контекст

Пользователь описал новый двухэтажный жилой дом площадью 180 м² и явно запросил полный набор конструктивных и инженерных разделов. Сессия №59 была создана заново из тех же документов, что и сессия №58. В ней подтверждённая геометрия кровельного покрытия равна 124,605 м², однако подбор нормативов не дал кандидатов. Для вентиляции система создала обязательную строку с количеством `1 компл` без документально подтверждённого объёма и также не нашла кандидатов. В результате в очереди проверки остаётся 50 строк, итог недоступен.

Сессия №58 и обычная смета №414 не являются образцом полноты: цена 3 154 397,72 ₽ не должна трактоваться как полный бюджет дома. Они не изменяются этой работой.

## Проблема

Сейчас в одном потоке смешаны три разных состояния:

1. Работа с доказанным объёмом, но без безопасно закреплённого норматива.
2. Требуемый пользователем элемент, для которого документов недостаточно, чтобы определить состав или объём.
3. Обычная позиция, которую действительно должен выбрать человек из кандидатов.

Для второго состояния планировщик создаёт условную позицию с количеством `1 компл`. Такая строка не может быть достоверно рассчитана и не даёт пользователю понятной границы состава. Для первого состояния контекст нормативов сохраняет общий набор кандидатов, но на этапе сопоставления повторно классифицирует намерение; подходящий ранее кандидат может исчезнуть. В интерфейсе число непокрытых элементов видно, но не видно, какие именно части не входят в стоимость и какие доказательства нужны.

## Цели

1. Смета содержит цену только по работам с доказанным объёмом, безопасным нормативом, полным составом ресурсов и актуальными ценами.
2. Неподтверждённые запрошенные компоненты показываются как явные границы комплектации, а не как фиктивные позиции для ручной проверки.
3. Нормативные кандидаты воспроизводимы: этап сопоставления использует только кандидатов, закреплённых именно за данной плановой работой.
4. AI-арбитр оценивает готовность и может запросить ограниченную адресную доработку, но не создаёт объёмы, материалы или состав работ без доказательств.
5. Обычная смета не может быть применена как полный бюджет, пока остаются неподтверждённые обязательные части. Результат в подтверждённой границе остаётся неприменяемым в обычную смету.

## Не-цели

- Не дополнять документы догадками о стропилах, утеплителе, инженерных трассах, приборах или отделке.
- Не подбирать нормы широким текстовым поиском после закрепления контекста.
- Не изменять сессию №58, смету №414, договоры или договорную логику.
- Не привязывать правила к региону, включая Татарстан.

## Выбранное решение

### 1. Доказательная граница комплектации

Планировщик создаёт сметную работу только при каноническом количестве, полученном из источника или разрешённого сценария с явно сохранённой основой. Если доказательного количества нет, он не создаёт строку `1 компл`. Вместо этого создаётся структурированный пробел комплектации:

```json
{
  "package_key": "ventilation",
  "work_key": "ventilation.air_exchange",
  "label": "Вентиляция: воздухообмен",
  "reason": "ventilation_duct_takeoff_missing",
  "required_evidence": "План вентиляции или ведомость трасс, оборудования и воздухообменов",
  "status": "not_included"
}
```

Пробелы сохраняются в составе черновика и в снимке границы сметы. Они не превращаются в задания «выберите норму» и не участвуют в прямых затратах.

Сценарные количества допускаются только там, где действующий каталог уже объявляет их допустимыми; в таком случае снимок хранит основание и признак предварительности. Неподтверждённые инженерные системы, состав кровли и отделка не могут стать комплектами по умолчанию.

### 2. Закреплённые кандидаты по ключу работы

При формировании нормативного контекста каждое намерение получает стабильный `work_item_key`. Источник контекста сохраняет не только общий снимок каталога, но и отображение `work_item_key → candidate_ids` вместе с версией базы, хешем содержимого, ресурсным составом и ценовым покрытием.

На этапе сопоставления разрешены только кандидаты из этого отображения для конкретной работы. Повторная классификация может проверить совместимость, но не может расширить выборку, заменить ключ или потерять кандидата из-за иной интерпретации текста. Если безопасного кандидата нет, формируется структурированный пробел `normative_candidate_missing` с причиной и доказательной основой, а не пустая строка очереди.

Полный ресурсный состав выбранной нормы передаётся без удаления самостоятельных ресурсов, машин и труда. Норма доступна только когда подтверждены её версия, применимость, ресурсы и цены.

### 3. Готовность и AI-арбитр

Готовность разделяется на два конечных состояния:

| Состояние | Условие | Действие пользователя |
| --- | --- | --- |
| `ready_for_application` | Все обязательные части рассчитаны и проверены | Можно применить как обычную смету |
| `confirmed_scope_only` | Рассчитанные позиции безопасны; все нерассчитанные обязательные части описаны структурированными пробелами | Можно изучить и дополнить источники; применение как полного бюджета недоступно |

Существующий AI-арбитр остаётся ограниченным по токенам и схеме ответа. Он получает только доказательства и снимок границы, затем выбирает: подтверждение, ограниченную адресную доработку, подтверждённую границу или обязательную проверку человеком. Адресная доработка допускается только для указанного пакета, имеет лимит попыток и принимается лишь после детерминированной проверки. Отсутствие ответа AI, исчерпание лимита или некорректный ответ переводят черновик в безопасный статус проверки человеком.

### 4. Интерфейс

В панели границы сметы:

- для `confirmed_scope_only` показывается заголовок «Смета готова в подтверждённой границе»;
- отдельный блок «Не включено в стоимость» перечисляет разделы, компонент, причину и требуемый источник;
- очередь проверки содержит только реальные решения: подтверждённый объём и закреплённые кандидаты нормы;
- прямые затраты, накладные расходы, сметная прибыль и коммерческий бюджет выводятся раздельно; отсутствующие показатели не маскируются нулём;
- вместо кнопки применения выводится понятное объяснение: «Добавьте подтверждающие данные, чтобы подготовить полный бюджет».

Пользователь с любыми правами просмотра может увидеть границу. Изменение источников, адресная доработка и применение остаются в действующем контуре прав.

## Контракт снимка границы

Существующий снимок расширяется обратносуместимыми полями:

```ts
type ScopeGap = {
  package_key: string;
  work_key: string;
  label: string;
  reason: string;
  required_evidence: string;
  status: 'not_included' | 'needs_normative_candidate';
};

type EstimateScopeBoundary = {
  claim_status: 'full_scope' | 'confirmed_scope_only';
  scope_gaps: ScopeGap[];
  priced_direct_cost: number | null;
  overhead_cost: number | null;
  estimated_profit: number | null;
  commercial_budget: number | null;
};
```

Старые снимки без `scope_gaps` продолжают отображаться через существующие предупреждения. Новый поток не создаёт фиктивную позицию как замену пробела.

## Проверки

1. Unit: планировщик не создаёт работу с `planner_fallback`; вместо неё создаёт структурированный пробел.
2. Unit: контекст нормативов хранит кандидатов по `work_item_key`; сопоставление использует закреплённый набор и не запрашивает широкий каталог.
3. Unit: отсутствие закреплённого кандидата создаёт пробел, а не неразрешимую строку проверки.
4. Unit/feature: состояние `confirmed_scope_only` требует валидности всех рассчитанных позиций и запрещает применение как полного бюджета.
5. Frontend: панель показывает понятные названия пробелов, причину и требуемое подтверждение; действие применения отсутствует для неполной границы.
6. Регрессия: сессия №59 после пересоздания не содержит строк с количеством `1 компл` без источника; кровельное покрытие с подтверждённым количеством либо имеет закреплённого кандидата, либо отображается как конкретный нормативный пробел.

## Критерии выпуска

После проверки в изолированных тестах и развёртывания создаётся новая версия сессии №59 штатным AI-потоком. Она остаётся неприменённой. Результат должен быть либо готовой обычной сметой при полном доказательном покрытии, либо конечной «Сметой в подтверждённой границе» с прозрачным перечнем исключений и без ложного полного итога. Ни один вариант не допускает выдуманных объёмов, материалов или ресурсов.

## Выбор источника цен

Если для объекта доступно несколько активных источников цен, МОСТ не подменяет выбор пользователя и не заимствует его из другой сметы. На шаге «Объект» пользователь выбирает действующий источник, после чего штатно запускается пересборка той же AI-сессии. Сервер повторно проверяет активность источника, сохраняет его для сессии и не затрагивает закреплённый набор нормативов. Недоступный источник не принимается.
