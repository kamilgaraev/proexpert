# Полная переработка AI-сметчика МОСТ — дизайн

**Дата:** 2026-07-11  
**Статус:** утверждено пользователем  
**Область:** `EstimateGeneration`, связанный интерфейс МОСТ и управление модулем в Filament

## 1. Цель

Переработать AI-сметчик МОСТ в production-ready инструмент, который принимает описание объекта, документы, чертежи, изображения и ручные наброски, строит проверяемую модель объекта, извлекает доказуемые объемы, подбирает нормативы и цены, формирует детерминированный черновик и передает его сметчику для контролируемого применения.

AI-сметчик остается помощником сметчика. Он не скрывает неопределенность, не создает несуществующие нормы и не рассчитывает финансовые итоги языковой моделью.

## 2. Граница изменений

Вся внутренняя реализация `EstimateGeneration` находится в области переработки. Разрешены:

- изменение и удаление старых AI-маршрутов, сервисов, jobs, UI-компонентов и тестов;
- изменение, пересоздание и очистка таблиц, принадлежащих только AI-сметчику;
- удаление тестовых AI-сессий и связанных с ними внутренних данных;
- замена существующего workflow без feature flags и runtime fallback;
- расширение Filament-раздела AI-сметчика.

Обычные сметы МОСТ находятся вне области изменений:

- существующие обычные сметы, позиции, версии и пользовательские данные не мигрируются и не изменяются;
- модели и бизнес-процесс обычных смет не рефакторятся в рамках этой программы;
- AI-модуль может только создать новую обычную смету через один защищенный application use case;
- после создания AI-модуль не обновляет и не удаляет обычную смету;
- повторное применение одной AI-сессии не создает вторую смету;
- удаление или архивирование AI-сессии не влияет на созданную обычную смету.

## 3. Стратегия поставки

Используется clean cutover внутри существующего модуля `EstimateGeneration`:

- новый код заменяет старый контур по мере готовности каждого из четырех планов;
- старые классы, маршруты, типы и тесты удаляются вместе с их заменой;
- feature flags, двойные API, параллельный legacy workflow и fallback-реализации не создаются;
- эксплуатационная обратимость обеспечивается атомарными Git-коммитами, резервной копией и откатом деплоя;
- каждый план оставляет модуль в рабочем и тестируемом состоянии;
- работа выполняется непосредственно в `main` короткими Conventional Commits.

## 4. Архитектура

### 4.1 Workflow

Управляет сессией, статусами, разрешенными действиями, правами, повторными запусками и аудитом. Только workflow service может менять состояние сессии.

### 4.2 Document Intelligence

Принимает файлы и извлекает текст, таблицы, векторную геометрию, визуальные элементы и факты. Каждый результат содержит источник, метод получения, confidence и версию обработчика.

### 4.3 Estimate Intelligence

Формирует предполагаемый состав работ и объемы, ищет нормативных кандидатов, ранжирует их и создает очередь спорных решений. AI выбирает только из существующих данных и не рассчитывает деньги.

### 4.4 Deterministic Draft

Собирает локальные сметы и позиции, проверяет единицы, применяет нормативные ресурсы и региональные цены. Все количества ресурсов, коэффициенты, округления и суммы рассчитываются обычным кодом.

### 4.5 Review & Apply

Предоставляет сметчику блокировки, предупреждения и спорные позиции. После подтверждения создает новую обычную смету через `ApplyGeneratedEstimate`. Это единственная точка записи AI-модуля в обычные сметы.

## 5. Статусная модель

Сессия AI-сметчика является корневым агрегатом.

| Статус | Значение |
| --- | --- |
| `draft` | Ввод и документы можно менять |
| `processing_documents` | Документы обрабатываются |
| `input_review_required` | Требуется решение по входным данным |
| `ready_to_generate` | Данных достаточно для генерации |
| `generating` | Формируется черновик |
| `estimate_review_required` | Есть спорные позиции |
| `ready_to_apply` | Все блокирующие проверки пройдены |
| `applying` | Создается обычная смета |
| `applied` | Обычная смета создана, AI-сессия неизменяема |
| `failed` | Терминальная техническая ошибка текущей попытки |
| `cancelled` | Обработка отменена пользователем |
| `archived` | Историческая сессия скрыта из рабочего списка |

Backend возвращает для каждого состояния:

- `available_actions`;
- `blocking_issues`;
- `warnings`;
- `next_action`;
- `processing_stage`;
- `processing_progress`;
- `state_version` для optimistic locking.

Контроллеры и jobs не выставляют статус напрямую.

## 6. Поток данных и доказательность

```text
Документ или изображение
  -> страница, таблица или геометрическая область
  -> распознанный элемент или факт
  -> объем с evidence
  -> предполагаемая работа
  -> нормативные кандидаты
  -> выбранная норма и ресурсы
  -> региональная цена
  -> детерминированный черновик
  -> очередь проверки
  -> готовность
  -> новая обычная смета
```

Каждый объем хранит:

- document и page ID;
- область страницы, координаты или диапазон ячеек;
- исходное значение и единицу;
- преобразование и коэффициент;
- метод извлечения;
- confidence;
- версию алгоритма или модели;
- признак `evidenced`, `user_confirmed` или `estimated`.

Каждое нормативное решение хранит исходный запрос, кандидатов, оценки, hard gates, причину выбора, версию нормативной базы и решение пользователя.

Изменение исходного документа инвалидирует все зависимые результаты. Пользовательские исправления добавляются как отдельные решения и не затирают исходный результат.

## 7. Чертежи, планы, изображения и наброски

Графический контур является обязательной частью первой production-ready версии.

### 7.1 Поддерживаемые источники

- векторные проектные PDF;
- растровые PDF и сканы;
- PNG и JPEG;
- фотографии бумажных планов;
- ручные наброски домов и помещений;
- DWG и DXF.

### 7.2 Каскад обработки

1. Классифицировать лист: план, фасад, разрез, узел, инженерная схема, экспликация, спецификация или свободный набросок.
2. Для PDF/DWG/DXF извлечь линии, полилинии, слои, блоки, размеры, масштаб и текст.
3. Для растра выполнить выравнивание, устранение перспективного искажения, очистку и оценку качества.
4. Vision-модель обнаруживает стены, помещения, двери, окна, лестницы, этажи, кровлю, фундамент, инженерные точки, трассы, размеры и подписи.
5. Объединить результаты в нормализованную модель здания.
6. Показать пользователю визуальную проверку распознанной геометрии до генерации.

### 7.3 Нормализованная модель здания

Содержит:

- этажи и помещения;
- наружные и внутренние стены;
- проемы;
- поверхности;
- конструктивные элементы;
- инженерные элементы и трассы;
- связи между элементами;
- evidence и confidence для каждого объекта.

### 7.4 Масштаб и точность

Геометрические объемы разрешено считать только при наличии:

- масштаба документа;
- распознанной размерной линии;
- либо контрольного размера, подтвержденного пользователем.

Без масштаба система формирует только оценочные объемы и обязана запросить минимальные параметры объекта.

### 7.5 Ручной набросок

Набросок с размерами используется для построения измеряемой модели. Для наброска без размеров система запрашивает общую площадь или габарит, этажность, высоту, материалы стен, фундамент, кровлю, уровень отделки и регион.

Результат содержит диапазон объемов, допущения и обязательные позиции проверки. Система не выдает набросок без масштаба за точный рабочий чертеж.

### 7.6 Пользовательская проверка

Пользователь может подтвердить масштаб, исправить помещение, назначение, границу, этажность, проем, материал и ошибочно распознанный элемент.

## 8. Нормативы, цены и AI

- Нормативный retrieval использует вид работ, материал, технологию, конструкцию, единицу, раздел и тип объекта.
- Hard gates исключают несовместимые единицы, материалы и виды работ.
- LLM используется только как ограниченный reranker существующих кандидатов.
- Модель не может создать нормативный код или обойти hard gate.
- При timeout или некорректном ответе операция становится диагностируемой ошибкой; постоянный fallback-алгоритм не сохраняется.
- Каждая цена имеет регион, зону, период, источник, версию и snapshot расчета.
- Цена другого региона или периода не используется молча.
- Изменение региона инвалидирует зависимый ценовой результат.

## 9. Review и quality gates

Применение запрещено, если:

- не подтвержден масштаб использованного чертежа;
- есть критический конфликт геометрии;
- платная позиция не имеет доказанного или подтвержденного объема;
- отсутствует нормативное решение;
- несовместимы единицы;
- отсутствует версия цены;
- оценочный объем требует подтверждения;
- обнаружен потенциальный дубль;
- не закрыты blocking review items.

Предупреждение можно принять только явным действием сметчика. Решение записывается в аудит.

## 10. Ошибки и фоновые операции

Ошибки делятся на:

- восстанавливаемые: временная недоступность провайдера, timeout, обрыв S3, ошибка отдельной страницы;
- требующие пользователя: нет масштаба, конфликт размеров, нечитаемый документ, неизвестный материал;
- терминальные технические: поврежденный файл, нарушение контракта, исчерпанные retry.

Jobs используют idempotency key, optimistic locking и checkpoints. Повторная доставка не создает дублей. Частичный сбой страницы не отменяет успешные страницы. Повторно обрабатываются только неуспешные единицы работы.

## 11. API и frontend

Frontend получает бизнес-решения от backend и не вычисляет `canApply` самостоятельно.

Основные ресурсы API:

- sessions;
- session snapshot/status;
- documents и processing units;
- normalized building model;
- generated packages и work items;
- review queue и decisions;
- benchmark/quality summary;
- apply, retry, cancel и archive actions.

Новый workspace состоит из шагов:

1. параметры объекта;
2. документы и изображения;
3. проверка геометрии;
4. структура и объемы;
5. черновик сметы;
6. спорные позиции;
7. итог и применение.

Текущий монолитный `EstimateGenerationWorkspacePage.tsx` удаляется после переноса. Новый route-level компонент не содержит бизнес-вычислений и делегирует зоны отдельным feature-компонентам, hooks и typed services.

## 12. Filament: центр управления AI-сметчиком

Существующий раздел обучающих наборов расширяется в единую навигационную группу «AI-сметчик».

### 12.1 Обзор

Dashboard показывает сессии, статусы, длительность, p95, очереди, зависшие задания, качество, review rate, применение результатов, токены и стоимость за выбранный период.

### 12.2 Сессии

Реестр и карточка сессии показывают организацию, проект, пользователя, timeline стадий, документы, позиции, проблемы, модели, версии, стоимость, jobs и audit events.

Разрешенные действия: retry упавшей стадии, отмена зависшей сессии, повторная постановка job, перевод документа на ручную проверку, архивирование тестовой сессии и формирование диагностического отчета.

Filament не изменяет и не удаляет обычные сметы.

### 12.3 Usage ledger

Каждый внешний AI/OCR/vision-вызов хранит:

- correlation, session, document и page ID;
- организацию и проект;
- стадию, провайдера и модель;
- input, cached input, output и reasoning tokens;
- изображения, detail и страницы OCR;
- длительность и попытки;
- результат;
- snapshot тарифа и рассчитанную стоимость.

Историческая стоимость не пересчитывается после изменения тарифа.

### 12.4 Ошибки и очереди

Ошибки группируются по fingerprint и фильтруются по периоду, организации, стадии, провайдеру и модели. Prompt, секреты, персональные данные и содержимое документов не отображаются.

Мониторинг показывает ожидающие, выполняющиеся, повторные, зависшие и окончательно упавшие jobs, возраст старейшей job и время стадий.

### 12.5 Обучение и benchmark

Существующий `EstimateGenerationTrainingDatasetResource` управляет эталонами, типами наборов, утверждением примеров, версиями датасетов и результатами benchmark.

Наборы имеют назначение `development`, `regression` или `acceptance`. Закрытый acceptance-набор нельзя использовать для настройки алгоритмов.

### 12.6 Настройки и бюджеты

Filament управляет моделями, timeout и retry только для реально вызываемых AI-стадий: распознавание чертежей, классификация документов и подбор нормативов. Также управляются используемые runtime пороги classification/geometry/normative matching, лимиты и форматы документов, бюджеты и обязательная проверка при низкой уверенности. Детерминированные planning и pricing не имеют искусственных AI-настроек. API-ключи и секреты не показываются и не редактируются.

Каждое изменение настроек имеет permission и аудит старого и нового значения и применяется только к новым операциям.

## 13. RBAC

Пользовательский интерфейс получает отдельные permissions:

- `estimate_generation.view`;
- `estimate_generation.create`;
- `estimate_generation.upload_documents`;
- `estimate_generation.generate`;
- `estimate_generation.review`;
- `estimate_generation.select_normative`;
- `estimate_generation.export`;
- `estimate_generation.apply`.

Filament получает отдельные system-admin permissions для monitoring, retry/cancel, datasets, benchmarks, settings и budgets. Все resources объявляют явные authorization methods.

## 14. Метрики качества

Golden dataset включает векторные и растровые PDF, PNG/JPEG, фотографию бумажного плана, наброски с размерами и без них, DWG/DXF, многоэтажный объект, конфликтующие документы и низкое качество изображения.

Измеряются:

- классификация листов;
- ошибка масштаба;
- IoU помещений и стен;
- окна и двери;
- ошибка площади, периметра и объемов;
- полнота конструктивных и инженерных элементов;
- полнота работ;
- top-1 и top-3 нормативов;
- расхождение стоимости;
- review rate;
- стоимость и длительность обработки.

Целевые показатели:

- полнота типовых работ не ниже 90%;
- top-3 нормативного подбора не ниже 95%;
- 100% применяемых позиций имеют evidence;
- не менее 98% валидных сессий завершаются без технической ошибки;
- допустимый порог ошибки объемов фиксируется отдельно по типам документов на первом baseline run и после этого не расширяется без утверждения.

## 15. Тестирование

- Unit: workflow, units, evidence, hard gates, pricing, confidence и readiness.
- Feature: полный API workflow, RBAC, tenant isolation, idempotency, retry, cancel и apply.
- Contract: все payload, пагинация, optional fields, error shape и available actions.
- Queue: повторная доставка, частичная ошибка, timeout, зависание и ограничение памяти.
- Vision/CAD: golden dataset и benchmark.
- Frontend: Vitest и MSW для всех шагов и ошибок.
- Browser: gstack smoke, console, network и responsive состояния.
- Integration: AI-модуль создает новую обычную смету, не меняет существующие и не создает дубль при повторном apply.

## 16. Четыре плана реализации

### Plan 1. Workflow и RBAC

Статусный автомат, available actions, permissions, tenant isolation, idempotent apply, аудит и очистка старого workflow.

### Plan 2. Backend pipeline

Use cases, DTO, стадии, resumable jobs, evidence graph, session snapshot, usage ledger, ошибки, очереди и удаление старого orchestrator-кода.

### Plan 3. AI, чертежи и качество

Vision/CAD, нормализованная модель здания, объемы, normative retrieval/reranking, цены, learning datasets, benchmark, quality gates, стоимость моделей и удаление placeholder/недоказуемых эвристик.

### Plan 4. Frontend и Filament

Пошаговый workspace, визуальная проверка геометрии, review cockpit, итоговое применение, удаление монолитного экрана и полный Filament command center.

Порядок обязателен:

```text
Workflow/RBAC -> Backend pipeline -> AI/drawings/quality -> Frontend/Filament
```

## 17. Definition of Done

Программа завершена, когда:

- каждая применяемая позиция имеет проверяемое происхождение;
- графический контур работает на планах, изображениях и набросках;
- неопределенность видна и блокирует небезопасное применение;
- jobs восстанавливаются без дублей;
- права разделены по операциям;
- frontend не дублирует backend workflow;
- Filament показывает стоимость, токены, качество, ошибки, очереди и обучение;
- acceptance dataset подтверждает целевые метрики;
- обычные сметы не зависят от AI-сессии и не изменяются AI-модулем после применения;
- старые AI-классы, маршруты, UI и fallback-пути удалены;
- workflow и операторская документация МОСТ актуализированы.
