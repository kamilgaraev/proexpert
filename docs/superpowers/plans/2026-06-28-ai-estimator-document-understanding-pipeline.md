# AI-сметчик: document understanding и геометрия планировок

Дата: 2026-06-28

## Цель

Система должна принимать произвольные файлы по объекту и сама определять, что перед ней: планировка, чертеж, спецификация, ведомость, обычная проектная документация, эталонная смета или нерелевантный файл. После классификации она должна извлекать проверяемые факты и объемы, а затем собирать смету через нормативную базу и цены, с ручной проверкой только там, где уверенности не хватает.

## Текущее состояние

- Файлы загружаются в сессию AI-сметчика и проходят OCR/структурирование.
- В доменной модели уже есть сущности для фактов документа, элементов чертежа, объемов и выводов по области работ.
- Сохранение сметы теперь блокируется, если есть позиции без рассчитанной цены или безопасной нормы.
- Пользователь может выбирать норму/цену в мастере проверки; выбор записывается как обучающий пример.

Ограничение: одна загруженная PNG/PDF-планировка пока не превращается надежно в полноценную ведомость объемов уровня сметчика. Для этого нужен отдельный vision/CAD pipeline.

## Целевая архитектура

### 1. Intake classifier

На входе каждый файл получает `document_profile`:

- `document_role`: `floor_plan`, `architectural_drawing`, `engineering_drawing`, `specification`, `bill_of_quantities`, `reference_estimate`, `technical_document`, `photo_or_scan`, `other`.
- `source_format`: `pdf_vector`, `pdf_scan`, `image`, `dwg`, `dxf`, `xlsx`, `docx`, `unknown`.
- `confidence`, `page_roles`, `detected_scale`, `requires_manual_review`.

Классификатор должен быть multi-label: один PDF может содержать планы, узлы, спецификации и пояснительную записку.

### 2. Page understanding

Для каждой страницы:

- OCR-текст с координатами.
- Тип страницы: план, разрез, фасад, спецификация, текст.
- Найденные размеры, площади, помещения, оси, отметки, масштабы, легенды.
- Нормализованные bounding boxes и система координат страницы.

### 3. Drawing takeoff

Для планировок и чертежей:

- Определить масштаб: из штампа, размерных линий или калибровки по известному размеру.
- Выделить помещения, стены, перегородки, проемы, окна, двери, мокрые зоны, лестницы.
- Рассчитать площади, периметры, длины стен/перегородок, количество проемов, базовые объемы отделки.
- Сохранить `quantity_takeoff_candidates` с формулой, источником и уверенностью.
- Требовать ручной проверки, если масштаб, контур помещения или размерная линия не подтверждены.

### 4. Specification extraction

Для ведомостей и спецификаций:

- Парсить таблицы структурно, а не строковым OCR.
- Нормализовать строки в `scope_facts`, `material_facts`, `equipment_facts`, `quantity_takeoffs`.
- Сопоставлять позиции спецификаций с чертежными зонами и разделами сметы.

### 5. Estimator reasoning

После извлечения фактов:

- Слить данные из всех файлов в единый `project_understanding`.
- Разделить работы по локальным сметам.
- Для каждого объема подобрать нормативные работы ФСНБ/ФЕР/ТЕР и региональные цены.
- Отмечать допущения отдельно от доказанных объемов.
- Пропускать сохранение только через review master, если есть нерассчитанные или сомнительные позиции.

## MVP без BIM

Первый реалистичный этап без тяжелых BIM-моделей:

1. PNG/JPG/PDF планировок: классификация, OCR с координатами, извлечение подписей площадей и размеров.
2. PDF-сканы планов: распознавание помещений по текстовым площадям и размерным линиям.
3. PDF-векторные планы: извлечение линий, контуров и текста из векторного слоя.
4. XLSX/PDF спецификации: структурный импорт таблиц.
5. Review overlay: пользователь видит найденные помещения/размеры/объемы и подтверждает или исправляет.
6. Обучение: исправленные объемы, выбранные нормы и эталонные сметы идут в ranking/evaluation dataset.

DWG/DXF подключать вторым этапом через изолированный worker-конвертер в SVG/JSON/PDF, потому что это отдельная зона лицензий, безопасности и ресурсов.

## Что должна сделать команда

1. Собрать эталонный набор: реальные планировки PNG/PDF, DWG/DXF, спецификации, ведомости объемов и финальные сметы из Гранд-Сметы.
2. Для каждого объекта хранить связку: входные файлы, правильные объемы, выбранные нормы, итоговые цены, регион/период цен.
3. Разметить минимум 50-100 планировок: помещения, площади, размеры, стены, проемы, мокрые зоны, ошибки распознавания.
4. Выбрать технологию CAD-конвертации и проверить лицензионные ограничения для production.
5. Определить допустимую точность: например, площадь помещений до 3-5%, длины стен до 5-8%, количество проемов без пропусков критичных элементов.
6. Подготовить QA-набор: система должна сравниваться с эталонной сметой не только по сумме, но и по составу работ, нормам и объемам.

## Как кормить нейронку эталонными сметами

Сырые сметы не нужно просто отправлять в модель как текст. Их надо нормализовать:

- входной документ -> факты/объемы;
- строка эталонной сметы -> нормативная позиция, объем, цена, раздел работ;
- связь между найденным фактом и строкой сметы;
- решение пользователя в мастере -> положительный пример выбранной нормы;
- отклоненный вариант -> отрицательный пример.

Сначала эти данные должны использоваться для retrieval/ranking и автоматической оценки качества. До fine-tuning переходить только после накопления чистого датасета и стабильной схемы признаков.

## Критерии готовности

- Файл получает понятный тип и уверенность классификации.
- Планировка без другой документации дает проверяемый набор помещений, размеров и базовых объемов.
- Каждая сметная строка знает источник объема: документ, страница, область, формула.
- Нулевая цена или неподтвержденная норма блокируют сохранение сметы.
- Исправления пользователя сохраняются как обучающие примеры.
- Большая партия файлов обрабатывается через очереди и не держит web-request открытым.
