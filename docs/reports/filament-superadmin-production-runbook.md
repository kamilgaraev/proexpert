# Runbook Filament-суперадминки

Дата: 2026-05-27.
Ветка: `main`.
Панель: Filament `admin`, путь `/admin`, guard `system_admin`.

## Модель доступа

Суперадминка использует роли из `config/RoleDefinitions/system_admin/*.json`. Это отдельный контур от обычной админки: доступ проверяется через модель `SystemAdmin`, `SystemAdminRoleService`, `FilamentPermission`, политики ресурсов и `canView()` у виджетов.

Базовые условия входа:

- системный администратор активен;
- роль содержит `system_admin.access`;
- роль разрешает интерфейс `admin`;
- ресурс или виджет явно разрешен текущей роли.

## Роли

| Роль | Назначение |
| --- | --- |
| `super_admin` | Полное управление платформой и настройками. |
| `content_manager` | Блог, медиа, SEO, комментарии и шаблоны уведомлений. |
| `qa_engineer` | Проверка клиентских сценариев и операционных разделов без критичных прав. |
| `security_auditor` | Read-only аудит, проверки доступа, мониторинг и расследования. |
| `support_operator` | Обработка обращений, назначение ответственных и эскалации. |
| `support_viewer` | Наблюдение за обращениями без изменения данных. |

## Политика опасных действий

Критичные массовые удаления в Filament не используются. Для организаций, пользователей, подписок, модулей, поддержки и блога применяются доменные действия: приостановить, реактивировать, отменить, вернуть в работу, отправить в архив, восстановить черновик.

Опасное действие должно проходить через сервис, проверять permission, требовать подтверждение там, где меняет бизнес-состояние, и писать событие аудита.

## Аудит

События действий суперадминки пишутся через `SystemAdminAuditService` в существующее хранилище активности. Для расследования используются:

- раздел `Аудит`;
- фильтры по действию, субъекту, организации, периоду и уровню риска;
- маскирование чувствительных деталей в UI.

Отсутствие события аудита для критичного изменения считается дефектом.

## Организации

Основной сценарий проверки клиента:

1. Открыть `Организации`.
2. Найти организацию по названию, домену, владельцу или идентификатору.
3. Проверить статус, подписку, владельца, пользователей, модули, платежи и активность.
4. При необходимости использовать безопасные действия приостановки или реактивации.
5. После изменения проверить событие в `Аудит`.

## Биллинг

Платежные факты отображаются read-only. Изменение жизненного цикла подписки выполняется через сервисные действия: отменить, реактивировать, выдать или отозвать ручное продление. Не редактируйте неизменяемые поля платежа напрямую.

## Публикация статьи

1. Создать или открыть черновик в `Блог CMS`.
2. Заполнить заголовок, адрес, лид, тело статьи, автора, категорию, обложку и SEO.
3. Проверить редакционный чеклист и SEO-превью.
4. Использовать группу действий `Публикация`.
5. Для восстановления использовать историю версий только у черновиков.

Публикация не должна обходить обязательные пункты чеклиста.

## Уведомления

Для диагностики доставки откройте шаблон или журнал уведомлений, проверьте канал, статус, дату, количество попыток и описание ошибки. Тестовая отправка доступна только текущему системному администратору.

## Мониторинг

Раздел ошибок приложения показывает группу, важность, статус, количество повторов, последний раз, модуль, сообщение, адрес, метод и короткий путь файла. Служебные детали вроде полного стека и чувствительного контекста не выводятся в UI.

Статусы ошибок меняются отдельными действиями: отметить решенной, игнорировать, вернуть в работу. Изменение статуса пишет аудит.

## Проверки

Перед релизом выполнить:

```powershell
php artisan test tests\Feature\Filament --stop-on-failure
php artisan test tests\Unit\Filament --stop-on-failure
php artisan test tests\Unit\Blog --stop-on-failure
php artisan test tests\Unit\RoleDefinitions --stop-on-failure
vendor\bin\phpstan analyse app\Filament app\Services\Filament app\Services\Blog app\Services\Security app\Models\SystemAdmin.php --memory-limit=1G --no-progress
rg "DeleteBulkAction|ForceDeleteAction" app\Filament
```

Ожидание для `rg`: пустой вывод.

## Browser QA

Локальные URL `http://127.0.0.1:8000/admin`, `http://localhost:8000/admin`, `http://127.0.0.1/admin`, `http://localhost/admin` были недоступны. Dev-сервер не поднимался, потому что проектные правила запрещают запускать его только ради QA.

Когда будет доступен локальный или staging URL, выполнить ручной smoke по `docs/qa/filament-superadmin-checklist.md`.

## План отката

Откат выполняется стандартным revert соответствующего коммита в `main` с последующим прогоном целевых тестов. Если изменения уже развернуты, сначала проверить, затрагивал ли релиз миграции; локально миграции в рамках этой задачи не запускались.
