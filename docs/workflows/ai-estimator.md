# AI-сметчик МОСТ: эксплуатационный процесс

## Назначение и границы

AI-сметчик создаёт проверяемый расчётный черновик по описанию объекта, проектным документам, чертежам и рисункам. Результат остаётся внутри отдельной AI-сессии до явного применения. Модуль не редактирует существующие обычные сметы.

Единственная разрешённая граница записи в обычные сметы — `Application/Apply/LaravelGeneratedEstimateWriter`. Применение создаёт новую смету и её новые разделы, позиции и ресурсы. Повторное применение той же сессии возвращает уже созданную смету и не создаёт дубль. Отмена или архивирование AI-сессии не изменяет применённую смету.

## Участники

- Инициатор создаёт сессию, задаёт исходные параметры и загружает материалы.
- Сметчик проверяет распознанные данные, объёмы, нормы, ресурсы, цены и допущения.
- Руководитель проекта применяет согласованный черновик в новую обычную смету.
- Оператор наблюдает ошибки и попытки, повторяет упавший этап, отменяет или архивирует сессию в пределах выданных прав.
- Фоновые обработчики распознают документы и выполняют расчёт идемпотентно.

## Условия начала

- Пользователь аутентифицирован, выбрана организация и проект.
- Организация владеет проектом, а маршруты проверяют проектный контекст.
- У пользователя есть требуемое право для каждого действия.
- Для мутации передаётся актуальная `state_version`.
- Есть описание объекта или хотя бы один документ, чертёж либо рисунок.

## Основной сценарий

1. Создаётся сессия `draft` с версией состояния `0`.
2. Загружаются документы. Для каждого файла создаётся отдельная попытка распознавания.
3. Сессия переходит в `processing_documents`. Обработка извлекает текст, таблицы, геометрию, размеры, помещения, элементы и подтверждающие ссылки на источник.
4. Если исходные данные требуют вмешательства, сессия получает `input_review_required`. После исправления или подтверждения она становится `ready_to_generate`.
5. Запуск расчёта переводит сессию в `generating`. Активная попытка имеет собственный идентификатор; результат устаревшей попытки не публикуется.
6. Система строит состав работ, подбирает нормативы и ресурсы, рассчитывает цены и запускает проверку качества.
7. Спорный результат получает `estimate_review_required`. Полностью проверенный результат получает `ready_to_apply`.
8. Применение атомарно переводит сессию через `applying` в `applied` и создаёт ровно одну новую обычную смету.

## Рабочие действия API

Все пути находятся под `/api/v1/admin/projects/{project}/estimate-generation/sessions`.

| Действие | Метод и путь | Право |
| --- | --- | --- |
| Создать сессию | `POST /` | `estimate_generation.create` |
| Загрузить документы | `POST /{session}/documents` | `estimate_generation.upload_documents` |
| Начать анализ | `POST /{session}/analyze` | `estimate_generation.generate` |
| Подтвердить исходные данные | `POST /{session}/confirm-input` | `estimate_generation.review` |
| Запустить расчёт | `POST /{session}/generate` | `estimate_generation.generate` |
| Повторить упавший этап | `POST /{session}/retry` | `estimate_generation.generate` |
| Отменить сессию | `POST /{session}/cancel` | `estimate_generation.generate` |
| Архивировать сессию | `POST /{session}/archive` | `estimate_generation.generate` |
| Получить очередь проверки | `GET /{session}/review-items` | `estimate_generation.view` |
| Записать решение проверки | `POST /{session}/feedback` | `estimate_generation.review` |
| Найти норматив | `GET /{session}/normative-candidates/search` | `estimate_generation.select_normative` |
| Выбрать норматив | `POST /{session}/normative-candidate` | `estimate_generation.select_normative` |
| Пересобрать раздел | `POST /{session}/rebuild-section` | `estimate_generation.generate` |
| Экспортировать черновик | `GET /{session}/export` | `estimate_generation.export` |
| Применить черновик | `POST /{session}/apply` | `estimate_generation.apply` |

Мутации сессии принимают актуальную `state_version`. Ответ snapshot возвращает только разрешённые текущим статусом и правами действия вместе с HTTP-методом, путём и признаком подтверждения.

## Документы, чертежи и рисунки

Загрузка, повторная обработка или исключение материала после анализа инвалидирует вычисленные данные и активную попытку генерации. Сессия возвращается к обработке документов, очищает устаревшие результаты и повышает `state_version`. Завершившийся позднее обработчик со старым идентификатором попытки не может опубликовать результат.

Для чертежей и набросков система хранит связь вывода с документом и областью источника. Неполные размеры, конфликтующие масштабы, неподтверждённые объёмы и неуверенно распознанные элементы не превращаются незаметно в итоговые позиции: они попадают в очередь проверки.

## Проверка и повтор

Проверка охватывает исходные данные, объёмы, нормативы, ресурсы, цены и критические замечания качества. Исправление решения пересчитывает готовность сессии. Если после проверки обязательных замечаний нет, workflow переводит сессию в `ready_to_apply`; появление нового обязательного замечания возвращает её в `estimate_review_required`.

При ошибке сохраняются предыдущий активный статус и причина. Повтор из `failed` выполняется как отдельная операционная команда под tenant-scoped блокировкой:

- для обработки документов повторно ставятся только ожидающие и упавшие документы; исключённые документы не блокируют поток, а `needs_review` переводит сессию в `input_review_required`;
- явный повтор из `input_review_required` снова ставит `needs_review`, ожидающие и упавшие документы в обработку, исключая `ignored`; если обрабатывать нечего, описание или уже готовый документ переводят сессию в `ready_to_generate`, иначе она остаётся в понятном пользователю состоянии проверки;
- для генерации создаётся новый идентификатор попытки и после фиксации транзакции ровно один раз ставится задача расчёта;
- для применения сессия возвращается в `ready_to_apply`, чтобы пользователь безопасно повторил идемпотентный `apply`, а не осталась без работы в `applying`.

Каждая мутация использует оптимистическую блокировку: запрос с устаревшей версией отклоняется, а `state_version` повышается на каждое успешное изменение workflow.

Повторная доставка фоновой задачи и повторный запуск одной попытки не создают дубли документов, пакетов, результатов или обычных смет.

## Отмена и архивирование

Отмена — явное событие workflow для незавершённой сессии. Она переводит сессию в `cancelled`, не удаляя диагностические данные. Архивирование разрешено для `failed`, `cancelled` и `applied`; оно переводит сессию в `archived` и скрывает её из рабочего списка без изменения применённой сметы.

`applied`, `cancelled` и `archived` являются терминальными состояниями для обычных рабочих мутаций. Архивирование из `applied` и `cancelled` — отдельная операторская операция, разрешённая картой переходов.

## Расчётный конвейер

Расчёт выполняется только через `PipelineRunner`. Реестр содержит ровно девять последовательных этапов:

1. `understand_documents` — проверка состава и версий документов;
2. `understand_object` — построение модели объекта;
3. `extract_quantities` — извлечение и проверка объёмов;
4. `plan_work_items` — формирование состава работ;
5. `match_normatives` — подбор нормативов;
6. `assemble_resources` — сборка ресурсов;
7. `resolve_prices` — фиксация цен;
8. `build_draft` — построение расчётного черновика;
9. `validate_draft` — итоговая проверка и публикация результата.

Одна задача очереди исполняет не более одного этапа. Следующий этап определяется сохранённым графом зависимостей, а не переданным вызывающей стороной предположением. `analyze`, `generate`, `retry` и `rebuild-section` входят в тот же конвейер. Завершённый этап повторно не вычисляется при совпадающих версиях входа; изменение источника инвалидирует только зависимые результаты.

Каждый этап имеет обязательную контрольную точку, идентификатор физической попытки, ограниченный результат и приватный артефакт в S3. Общий размер артефактов одной попытки ограничен 8 МиБ. Публикация итогового черновика, переход workflow и запись события завершения выполняются атомарно.

## Очереди и восстановление

Horizon обслуживает три очереди соединения `redis_estimate_generation`:

- `estimate-generation` — документы, расчётные этапы и доставка итоговых событий;
- `estimate-generation-units` — страницы и другие независимые единицы документов;
- `estimate-generation-unit-maintenance` — восстановление истёкших аренд единиц.

Планировщик каждую минуту запускает восстановление единиц, восстановление расчётных конвейеров и доставку итоговых событий. Все три задания защищены от пересечения. Рабочая аренда этапа составляет не менее 2100 секунд, аренда единицы соответствует таймауту её обработчика; повторная доставка использует сохранённую попытку и не публикует устаревший результат. Итоговые события доставляются как минимум один раз, а бизнес-действие защищено стабильным ключом идемпотентности.

При расследовании оператор проверяет в таком порядке: актуальность `state_version` и идентификатора попытки, активную контрольную точку и срок аренды, состояние единиц документа, последнюю безопасную ошибку, событие завершения и подтверждение его доставки. Ручной запуск отдельного внутреннего сервиса этапа запрещён: повтор выполняется только через действие workflow или штатное восстановление.

## Учёт использования и ошибок

Каждая фактическая попытка обращения к AI, включая повтор и переключение модели, записывается отдельно. Журнал содержит организацию, сессию, этап, операцию, модель, измерения токенов или страниц, снимок тарифа, стоимость и итог попытки. Неизвестное измерение или цена сохраняются как неизвестные, а не как ноль.

Ошибки хранятся как неизменяемые события с безопасным кодом, категорией, отпечатком и ограниченным диагностическим контекстом. Тексты запросов и ответов, содержимое файлов, имена и пути файлов, ключи доступа и исходные сообщения исключений не сохраняются. Повтор закрывает только актуальную ошибку той же версии workflow.

История использования, ошибок, контрольных точек и подтверждений доставки живёт вместе с AI-сессией и удаляется только каскадно при её административном удалении. Архивирование историю не удаляет. Приватные S3-объекты документов и этапов должны обслуживаться lifecycle-политикой префикса `org-{organization_id}/ai-estimator/sessions/`; срок хранения не должен быть меньше срока хранения AI-сессии и её аудита.

## Snapshot и нагрузка чтения

`GET .../sessions/{session}/snapshot` возвращает согласованный операционный снимок из короткой read-only транзакции PostgreSQL. Сильный ETag учитывает workflow, документы, контрольные точки, единицы, evidence, использование, ошибки, итоговую доставку, пакеты и производные данные документов. Поддерживаются `If-None-Match`, слабое сравнение, список тегов и `*`; ответ `304` имеет пустое тело и приватные cache-заголовки.

Снимок имеет постоянный бюджет из 11 агрегирующих tenant-scoped запросов независимо от количества документов и результатов. Для восстановления и будущего административного интерфейса предусмотрены индексы по сессии, этапу, статусу, сроку аренды, версии источника, организации и времени записи.

## Миграции и PostgreSQL-контракты

Миграции Plan 1/2 применяются в порядке `000001`, `000100`, `000200`, `000300`, `000400`, `000500`, `000600`, `000900`. Они изменяют только таблицы AI-сметчика. До развёртывания выполняются opt-in PostgreSQL-наборы:

- конкуренция и неизменяемость контрольных точек, общий лимит артефактов;
- конкуренция, область организации, граф и инвалидирование evidence;
- идемпотентность, стоимость, ограничения и неизменяемость журнала использования;
- конкуренция, разрешение и неизменяемость истории ошибок;
- неизменяемость подтверждения доставки и каскадное удаление вместе с тестовой AI-сессией.

Локально эти тесты не запускаются без отдельного PostgreSQL-окружения и миграций. Их пропуск не считается подтверждением контракта БД.

## Граница с обычными сметами

Расчётный конвейер, восстановление, журналы и S3-артефакты не имеют права изменять таблицы обычных смет. Единственная граница — явное действие `apply` через `Application/Apply`, которое после проверки статуса и версии создаёт новую смету. Существующие сметы не обновляются и не используются как скрытый резервный путь. Общий AI Assistant не вызывает конвейер AI-сметчика и не зависит от него.

## Наблюдаемость и восстановление

Для расследования используются идентификаторы организации, проекта, сессии, документа и попытки, версия состояния, этап, прогресс и последняя ошибка. Оператор сначала определяет упавший этап и актуальность попытки, затем повторяет только этот этап. Ручное изменение полей workflow и прямой запуск записи в обычные сметы запрещены.
