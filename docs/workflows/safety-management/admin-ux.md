# Модуль «Охрана труда»: админский интерфейс

## Рабочее место

Админский экран должен быть операционным рабочим местом, а не отдельными несвязанными справочниками. Пользователь сначала видит общую картину, затем проваливается в проблемные зоны.

## Зоны экрана

- Дашборд: допущены, не допущены, частично допущены, истекающие документы, открытые нарушения, инциденты и просроченные действия.
- Наряды-допуски: список, статус, участники, причины блокировки, действия по активации и закрытию.
- Карточки ОТ сотрудников: требования, обучение, медосмотры и СИЗ.
- Проверки и замечания: шаблоны, обходы, замечания, связь с нарушениями.
- Документы: черновики журналов, карточек СИЗ и актов.

## Принципы отображения

- Причина блокировки должна быть видна без открытия каждого технического объекта.
- Статус допуска отображается как бизнес-статус, а не как набор сырых кодов.
- Пустое состояние объясняет, чего не хватает для начала работы.
- Опасные действия недоступны, если backend вернул блокеры.
- UI не пересчитывает требования, а показывает `admission_summary`, `problem_flags`, `blockers` и `warnings` из backend.

## Документы

Черновик документа отображается как структурированный набор данных: название, источник, период, участники, строки и предупреждения. Финальная печатная форма может быть подключена отдельным экспортным сервисом, но черновик уже должен быть пригоден для проверки специалистом ОТ.

## Ошибки

Сообщения должны быть бизнес-понятными: что не хватает, у кого проблема и какое действие нужно выполнить. В интерфейсе нельзя показывать внутренние термины `payload`, `dto`, `sql`, `constraint`, `exception`.
