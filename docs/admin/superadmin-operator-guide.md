# Операторский гайд Filament-суперадминки

Гайд описывает ежедневную работу в Filament-суперадминке МОСТ. Панель находится на `/admin`, использует guard `system_admin` и роли из `config/RoleDefinitions/system_admin`.

## Ролевая матрица

| Роль | Основные разделы | Основные действия |
| --- | --- | --- |
| `super_admin` | Все разделы | Полное управление платформой. |
| `content_manager` | Блог CMS, медиатека, SEO, уведомления | Создание и публикация статей, управление медиа, комментариями, SEO и шаблонами. |
| `qa_engineer` | Обзор, организации, пользователи, биллинг, модули, блог, уведомления | Проверка состояния платформы и клиентских сценариев без доступа к критичным настройкам. |
| `security_auditor` | Аудит, пользователи, организации, биллинг, модули, блог, уведомления | Read-only проверки доступа, событий и подозрительных изменений. |
| `support_operator` | Поддержка, организации, пользователи | Назначение и обработка обращений, внутренние заметки, эскалация. |
| `support_viewer` | Поддержка | Просмотр обращений без изменений. |

Если оператор не видит раздел в меню, сначала проверяется роль и permission в `config/RoleDefinitions/system_admin`, затем `shouldRegisterNavigation()` ресурса.

## Ежедневный обзор

1. Откройте `Обзор`.
2. Проверьте блоки здоровья платформы: активные организации, пробные периоды, платящие организации, просроченные подписки.
3. Проверьте риск-блоки: срочные обращения, ошибки платежей, риск-события аудита.
4. Перейдите в нужный рабочий раздел из навигации, не выполняя доменные действия из dashboard.

Dashboard предназначен для triage. Изменения выполняются в профильных resources, где есть подтверждения, audit trail и доменные проверки.

## Организации

Основной сценарий:

1. Откройте `Организации`.
2. Найдите организацию по названию, ИНН или статусу.
3. На detail-странице проверьте реквизиты, статус, подписку, метрики пользователей, проектов и хранилища.
4. Для ограничения доступа используйте действие приостановки. Для возврата доступа используйте реактивацию.

Правила:

- Не удаляйте организацию для операционного ограничения доступа.
- Приостановка и реактивация пишут audit event.
- Изменение подписки выполняется через раздел биллинга, а не через прямое редактирование реквизитов.

## Пользователи

Основные действия:

- блокировка пользователя;
- разблокировка пользователя;
- подтверждение email;
- отправка письма для сброса пароля.

Правила:

- Блокировка используется для ограничения входа, а не для удаления истории.
- Подтверждение email применяется только после проверки контекста пользователя.
- Сброс пароля отправляется через существующий notification flow и фиксируется в аудите.

## Биллинг

Разделы:

- `Тарифы`
- `Подписки`
- `Пакеты организаций`
- `Платежи`

Основные сценарии:

1. Для проверки клиента начните с организации и ее текущей подписки.
2. Для изменения жизненного цикла подписки используйте действия отмены, реактивации, выдачи или отзыва ручного продления.
3. Для проверки платежей используйте статус, дату операции и masked provider identifiers.

Правила:

- Платежные транзакции являются read-only поверхностью.
- Ручные продления требуют причины.
- Отмена и реактивация подписки пишут audit event.

## Модули и доступы

Разделы:

- `Модули`
- `Модули организаций`

Основные сценарии:

1. Проверьте модуль в каталоге: тип, модель оплаты, готовность, зависимости и конфликты.
2. Проверьте подключение модуля у организации: статус, источник, сроки и автопродление.
3. Используйте доменные действия для включения, продления, отключения и синхронизации прав.

Правила:

- Не обходите зависимости и конфликты модуля.
- Ручная синхронизация прав фиксируется в аудите.
- Массового удаления подключений нет.

## Блог CMS

### Создание статьи

1. Откройте `Блог CMS -> Блог`.
2. Создайте черновик.
3. Заполните заголовок, адрес, лид и основное полотно статьи.
4. Выберите автора, категорию, обложку и SEO-поля.
5. Сохраните черновик.

Форма подсказывает обязательные редакционные поля. Внутренние заметки свернуты, чтобы не мешать основному написанию.

### Публикация

1. Откройте статью в редакторе.
2. Проверьте редакционный чеклист.
3. Используйте группу `Публикация`:
   - опубликовать;
   - запланировать;
   - вернуть в черновик;
   - отправить в архив.
4. Если чеклист не готов, публикация будет заблокирована.

Правила:

- `Сохранить черновик` отделен от публикационных действий.
- Публикация не должна обходить чеклист.
- Архивирование требует подтверждения.
- Ревизии используются для проверки и восстановления черновиков.

### Медиатека

1. Загружайте изображения и документы только через `Медиатека`.
2. Для изображений заполняйте alt-текст.
3. Перед заменой файла проверьте, используется ли он в опубликованных статьях.

Правила:

- Удаление используемого медиафайла блокируется.
- Безопасная замена доступна только там, где сервис может обновить ссылки без повреждения опубликованных материалов.
- Неподдерживаемые типы файлов отклоняются.

## Уведомления

Разделы:

- `Уведомления`
- `Шаблоны уведомлений`
- `Диагностика доставки`

Рабочий сценарий:

1. Проверьте шаблон и preview на тестовых данных.
2. Используйте test-send только для текущего системного администратора.
3. При проблемах доставки откройте аналитику уведомлений и проверьте канал, статус, retry count и ошибку.

Правила:

- Шаблоны не удаляются без проверки использования.
- Тестовая отправка не должна уходить клиентам.
- Диагностика доставки предназначена для расследования, а не для ручного изменения истории.

## Поддержка

Основные действия:

- назначить оператора;
- изменить статус;
- добавить внутреннюю заметку;
- связать обращение с организацией;
- эскалировать.

Правила:

- Внутренние заметки не предназначены для клиента.
- Эскалация используется для срочных или рискованных обращений.
- Приватные технические поля видны только ролям с расширенными правами поддержки.

## Аудит

Когда использовать:

- проверка опасного действия;
- расследование жалобы;
- проверка доступа роли;
- поиск причины изменения статуса организации, подписки, пользователя, модуля или статьи.

Сценарий поиска:

1. Откройте `Аудит`.
2. Отфильтруйте по модулю, действию, организации, actor type или периоду.
3. Проверьте severity, result, actor, subject и correlation id.
4. При необходимости сопоставьте событие с профильным resource.

Правила:

- Audit resource read-only.
- Чувствительные детали маскируются.
- Отсутствие audit event для опасного действия считается дефектом.

## Политика опасных действий

Опасными считаются действия, которые меняют доступ, деньги, публикацию, права, поддержку или историю видимых клиенту данных.

Требования:

- действие должно быть разрешено role permission;
- действие должно идти через доменный сервис;
- для разрушительных операций нужно подтверждение;
- массового удаления организаций, тарифов, платежей, подписок и критичных связей нет;
- результат должен быть записан в audit trail;
- пользовательский текст должен быть понятным оператору и без внутренних технических терминов.

Контрольные команды:

```powershell
rg "DeleteBulkAction|ForceDeleteAction" app\Filament
php artisan test tests\Feature\Filament\SystemAdminDangerousActionsTest.php --stop-on-failure
```

## Инциденты

### Ошибка доступа

1. Проверьте активность системного администратора.
2. Проверьте роль в `config/RoleDefinitions/system_admin`.
3. Проверьте permission в `app/Filament/Support/FilamentPermission.php`.
4. Проверьте `shouldRegisterNavigation()` и policy ресурса.

### Ошибка публикации статьи

1. Откройте чеклист статьи.
2. Проверьте заголовок, адрес, категорию, автора, обложку, alt-текст, SEO и тело статьи.
3. Проверьте статус и дату планирования.
4. Если ошибка повторяется, смотрите unit/feature tests блога и audit events.

### Ошибка доставки уведомления

1. Откройте уведомление и relation manager диагностики.
2. Проверьте канал, статус, retry count и описание ошибки.
3. Проверьте шаблон и sample data.
4. Не меняйте историю доставки вручную.

### Жалоба на действие оператора

1. Откройте audit resource.
2. Найдите событие по actor, subject, organization или correlation id.
3. Проверьте роль оператора на момент действия.
4. Сопоставьте событие с доменным объектом.

## Финальная проверка

Перед релизом или крупным изменением суперадминки выполняются:

```powershell
php artisan test tests\Feature\Filament --stop-on-failure
php artisan test tests\Unit\Filament --stop-on-failure
php artisan test tests\Unit\Blog --stop-on-failure
php artisan test tests\Unit\RoleDefinitions --stop-on-failure
vendor\bin\phpstan analyse app\Filament app\Services\Filament app\Services\Blog app\Services\Security app\Models\SystemAdmin.php --memory-limit=1G --no-progress
rg "DeleteBulkAction|ForceDeleteAction" app\Filament
```

Browser QA выполняется только при уже доступном admin URL. Dev-сервер не поднимается только ради проверки. Если URL недоступен, это фиксируется в плане и итоговом отчете.
