# ERP-04 Next Steps

Дата: 2026-06-16

## Контекст

После слияния `PHERP-106`, `PHERP-107`, `PHERP-108` и `PHERP-109` в `main` коммерческий контур уже имеет базу по CRM, тендерам и коммерческим предложениям. Следующий бизнес-результат должен замкнуть переход из продажи в операционный проект.

## Рекомендованное направление

Идти дальше через `PHERP-112` и `PHERP-113`: конвертация сделки в проект и договор.

Причина: это создает рабочую цепочку `CRM / тендер / КП -> сделка -> проект -> договор`. После этого presale-бюджет и аналитика продаж будут опираться на реальные связи, а не на будущие допущения.

## Порядок работ

1. `PHERP-112`: спроектировать wizard и API-контракт конвертации сделки в проект и договор.
2. `PHERP-113`: реализовать backend transaction flow конвертации.
3. Добавить admin UI wizard: preview, недостающие поля, ошибки, результат с переходами в созданный проект и договор.
4. `PHERP-110` и `PHERP-111`: спроектировать и реализовать перенос presale-сметы в проектный бюджет.
5. `PHERP-114` и `PHERP-115`: спроектировать и реализовать sales analytics / tender-to-project dashboard.

## Критические требования к конвертации

- транзакционность создания связанных сущностей;
- idempotency key, чтобы повторный запуск не создавал дубли;
- preview до фактического создания объектов;
- понятные ошибки валидации до начала транзакции;
- audit trail по исходной сделке, созданному проекту и договору;
- права доступа с учетом коммерческой конфиденциальности;
- связи с CRM, тендером, КП, контрагентом, будущим бюджетом и документами.

## Следующая задача

Начать с `PHERP-112`: не писать сразу production flow, а сначала зафиксировать wizard, данные, API-контракт, права, ошибки и idempotency/rollback требования для `PHERP-113`.
