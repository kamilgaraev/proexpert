# Модель конфликтов обмена ProHelper и 1С

Дата: 2026-06-07
Задача: PHERP-72
Статус документа: аналитика для проверки

## Контекст

Документ задает целевую модель конфликтов обмена ProHelper и 1С. Конфликт отличается от технической ошибки: retry не должен автоматически повторять сообщение, пока владелец данных не примет решение.

Границы:

- ProHelper остается операционным source of truth.
- 1С остается бухгалтерским и налоговым source of truth.
- Операционные статусы ProHelper не равны учетным статусам 1С.
- ProHelper не становится бухгалтерским ядром.
- Raw payload, stack trace, токены и секреты нельзя показывать пользователям.

## Что считается конфликтом

Конфликт возникает, когда система не может безопасно решить, какое значение, связь или статус использовать.

Это не конфликт:

- временная недоступность 1С;
- сетевой timeout без расхождения данных;
- отсутствие маппинга, если нет конкурирующего кандидата;
- бизнес-отказ 1С с однозначной причиной валидации.

Это конфликт:

- один external id указывает на другой local object;
- local object уже связан с другим external id;
- 1С вернула duplicate с отличающейся суммой, датой или контрагентом;
- документ ProHelper изменился после отправки, а 1С уже приняла старую версию;
- 1С пытается изменить поле, которым владеет ProHelper;
- ProHelper пытается повторно отправить документ, который 1С уже провела с отличающимися контрольными полями;
- payroll-source package изменился после lock;
- складское движение в 1С отражено в другой партии/складе, чем ожидал ProHelper.

## Типы конфликтов

| Тип | Пример | Автоматическое решение |
| --- | --- | --- |
| `mapping_duplicate` | один контрагент 1С найден для двух подрядчиков | нет |
| `mapping_missing_with_candidates` | нет active mapping, но есть несколько близких кандидатов | нет |
| `external_id_mismatch` | local document связан с другим external id | нет |
| `owner_field_mismatch` | 1С вернула другую сумму или контрагента | нет |
| `status_regression` | 1С вернула rejected после posted или conflicting callback | нет |
| `stale_source` | source hash изменился после отправки | нет |
| `line_items_mismatch` | отличаются строки акта, заказа, склада или payroll package | нет |
| `amount_or_vat_mismatch` | отличается сумма, НДС или валюта | нет |
| `date_or_period_mismatch` | отличается дата документа, период акта или payroll period | нет |
| `posted_edit_attempt` | попытка редактировать учетно проведенный документ без корректировки | нет |
| `ambiguous_timeout_duplicate` | после timeout найден похожий документ 1С, но контрольные поля не совпали | нет |
| `ownership_violation` | внешняя система пытается поменять операционный workflow ProHelper | нет |

## Severity

| Severity | Значение | Пример |
| --- | --- | --- |
| `info` | Не блокирует обмен, требует внимания позже | 1С вернула дополнительный внешний номер |
| `warning` | Можно продолжить часть процесса, но нужна сверка | название контрагента отличается при совпавшем ИНН |
| `blocking` | Блокирует сообщение или документ | сумма платежа отличается |
| `critical` | Блокирует scope или требует поддержки | duplicate posted document, нарушение схемы ownership |

Для платежей, payroll и складского учета расхождения суммы, периода, контрагента, сотрудника, склада или материала должны быть минимум `blocking`.

## Status

| Статус | Значение |
| --- | --- |
| `open` | Конфликт создан |
| `in_review` | Владелец данных начал разбор |
| `waiting_external` | Нужна проверка в 1С, банке или ЗУП |
| `waiting_prohelper_change` | Нужна правка данных ProHelper |
| `resolved` | Решение принято, можно продолжать |
| `deferred` | Отложено с причиной и сроком |
| `stale` | Конфликт устарел из-за новой версии источника |
| `closed_as_obsolete` | Закрыт как неактуальный |
| `dead_letter_linked` | Сообщение переведено в dead-letter до ручного решения |

Закрытие без решения должно быть возможно только с причиной и audit trail.

## Owner

| Контур | Owner |
| --- | --- |
| Контрагенты и реквизиты | MDM steward + бухгалтерия |
| Проекты и ЦФО | PMO + финконтроль |
| Договоры и акты | договорной отдел + бухгалтерия |
| Платежи и банковские факты | финансовый менеджер + бухгалтерия |
| Закупки | снабжение + бухгалтерия при учетном документе |
| Склад | склад/логистика + бухгалтерия по учетному складу |
| Payroll-source | HR + зарплатный бухгалтер |
| Технический schema/authorization | поддержка интеграции |

Owner определяет, кто видит действие "принять ProHelper value" или "принять 1С value". Поддержка не должна принимать бизнес-решение вместо владельца данных, если это меняет учетный смысл.

## Допустимые действия

| Действие | Кто может | Ограничения |
| --- | --- | --- |
| Принять значение ProHelper | data owner scope + integration manager | нельзя перезаписать бухгалтерский `posted` без корректировки |
| Принять значение 1С | бухгалтерия/финконтроль/HR по scope | нельзя менять операционный workflow, если владелец ProHelper |
| Связать вручную | mapping manager scope | требуется validation и audit |
| Создать новый mapping | mapping manager или MDM steward | duplicate check обязателен |
| Отправить корректировку | владелец документа | только для документов, где корректировка допустима |
| Requeue message | integration manager | только после resolved и актуального source hash |
| Отложить | owner или integration manager | нужен срок и причина |
| Закрыть как неактуальный | owner или support по техническим конфликтам | нужна проверка stale/current state |
| Перевести в dead-letter | integration support | не решает конфликт, только останавливает автоматическую обработку |

## Auto/manual resolution

Автоматическое решение допустимо только для низкорисковых случаев:

- повторный callback с тем же статусом и теми же контрольными полями;
- accepted duplicate с полностью совпавшим idempotency key;
- обновление external display name при совпавшем immutable external id и отсутствии owner-field изменений.

Ручное решение обязательно для:

- суммы, НДС, валюты;
- контрагента, организации, проекта, договора;
- payroll employee mapping и source rows;
- warehouse/material/quantity/party mismatch;
- already posted documents;
- любых конфликтов с несколькими кандидатами.

## Audit trail

Каждый конфликт должен хранить:

| Поле | Назначение |
| --- | --- |
| `id` | Идентификатор конфликта |
| `organization_id`, `project_id`, `one_c_base_id` | Контур |
| `scope`, `conflict_type`, `severity`, `status` | Классификация |
| `source_object_type`, `source_object_id` | Объект ProHelper |
| `external_object_type`, `external_id` | Объект 1С |
| `message_id`, `operation_id`, `dead_letter_id` | Связь с exchange |
| `prohelper_value_preview`, `external_value_preview` | Безопасное сравнение |
| `source_hash`, `external_hash` | Защита от stale resolution |
| `owner_role`, `assigned_to_user_id` | Ответственный |
| `resolution_action`, `resolution_reason` | Принятое решение |
| `resolved_by_user_id`, `resolved_at` | Кто и когда решил |
| `created_at`, `updated_at` | Временные метки |

Audit должен хранить историю изменений статуса, комментарии, выбранные значения и affected messages.

## Stale conflict protection

Перед применением решения нужно проверить:

- source object все еще существует и принадлежит организации;
- `source_hash` совпадает с текущей версией или решение явно применимо к новой версии;
- mapping не был изменен другим пользователем;
- message не был уже accepted/posted другим attempt;
- external object не изменил контрольные поля;
- payroll period/package не изменил source hash;
- складское движение не было скорректировано новым движением.

Если проверка не проходит, конфликт получает `stale`, а пользователь видит безопасный текст: "Данные изменились, откройте конфликт заново".

## Связь с retry и dead-letter

Правила:

- конфликт блокирует автоматический retry;
- retry после решения создает новый attempt с тем же idempotency key только если business payload не менялся;
- если payload изменился, создается новая message version;
- dead-letter может ссылаться на конфликт, но не заменяет его;
- закрытие dead-letter без решения конфликта запрещено;
- `resolved` conflict должен явно указывать affected messages для requeue.

## UI карточка конфликта

Карточка должна показывать:

- короткий статус и понятную причину;
- объект ProHelper и объект 1С;
- сравнение значений в две колонки;
- owner и SLA;
- связанные messages, attempts, mappings, documents;
- безопасный next action;
- историю решений;
- кнопки только доступных действий.

Нельзя показывать:

- raw payload;
- stack trace;
- токены и секреты;
- полные банковские реквизиты без отдельного права;
- персональные/payroll данные сверх разрешенного preview.

Примеры безопасных пользовательских текстов:

| Сценарий | Текст |
| --- | --- |
| Сумма платежа отличается | "Сумма платежа в 1С отличается от суммы в ProHelper. Требуется сверка перед повторной отправкой." |
| Нет надежного маппинга | "Найдено несколько похожих объектов 1С. Выберите корректное сопоставление." |
| Документ уже проведен | "Документ уже отражен в 1С. Изменение возможно только через корректировку." |
| Данные устарели | "Данные изменились после создания конфликта. Обновите карточку и повторите проверку." |

## API будущего conflict registry

Базовый префикс: `/api/v1/admin/one-c-exchange/conflicts`.

| Endpoint | Назначение |
| --- | --- |
| `GET /` | Список конфликтов |
| `GET /{id}` | Карточка конфликта |
| `POST /{id}/assign` | Назначить ответственного |
| `POST /{id}/accept-prohelper` | Принять значение ProHelper |
| `POST /{id}/accept-external` | Принять значение 1С |
| `POST /{id}/link-mapping` | Связать вручную |
| `POST /{id}/create-correction` | Создать корректирующее действие |
| `POST /{id}/defer` | Отложить |
| `POST /{id}/close-obsolete` | Закрыть как неактуальный |
| `POST /{id}/requeue` | Вернуть связанные сообщения после решения |
| `GET /{id}/audit` | История |

Пример resolution request:

```json
{
  "reason": "В 1С найден корректный контрагент, совпали ИНН и КПП",
  "expected_message_action": "requeue_blocked_messages",
  "confirm_source_hash": "sha256:..."
}
```

Ответ должен включать safe message, новый status, affected messages и недоступные действия с причинами.

## Открытые вопросы

- Нужен ли dual approval для принятия значения 1С в платежах, payroll и складских расхождениях.
- Какие конфликтные поля можно разрешать массово, а какие только по одной карточке.
- Какой SLA нужен для blocking и critical конфликтов.
- Нужно ли связывать conflict registry с MDM duplicate groups как единый пользовательский queue.
