# Отказы, retry и актуальность обмена с 1С

Дата: 2026-06-07
Задача: PHERP-63
Статус документа: аналитика для проверки

## Контекст

Документ задает целевые правила обработки отказов, повторных попыток и контроля актуальности для обмена ProHelper и 1С.

Границы:

- ProHelper остается операционной ERP.
- 1С остается учетным source of truth.
- текущий `OneCBasicExchange` является ручным MVP с токенами, маппингами, import/export и историей запусков;
- production-контур должен добавить очереди, retry, dead-letter, безопасные ошибки, freshness metrics и reconciliation.

## Принципы

- Любая попытка обмена должна иметь idempotency key.
- Технический сбой и бизнес-отказ 1С обрабатываются по-разному.
- Повторная отправка не должна создавать дубли документов в 1С.
- Ошибки для пользователя должны быть понятными и без внутренних секретов.
- Полные вложения, токены, пароли, банковские секреты и технические трассировки не пишутся в пользовательский текст ошибки.
- Если проблема связана с маппингом или конфликтом данных, автоматический retry не выполняется до ручного решения.
- Актуальность измеряется по каждому scope, direction и organization.

## Типы отказов и правила retry

| Сценарий | Признак | Retry policy | Dead-letter / review | Статус пользователю | Действие поддержки |
| --- | --- | --- | --- | --- | --- |
| 1С недоступна | connection refused, DNS, недоступен endpoint | автоматический retry с backoff: 1 мин, 5 мин, 15 мин, 1 час, 3 часа | после 5 попыток в `dead_letter` | "1С временно недоступна, повторим отправку" | проверить доступность базы/обработки/сети |
| Timeout | нет ответа за лимит времени | retry только с тем же idempotency key; перед повтором проверить, не принят ли документ | после 5 попыток или при неоднозначном ответе | "Ожидаем подтверждение от 1С" | сверить по correlation/idempotency key |
| Validation error | 1С вернула отказ по данным | автоматический retry запрещен | `rejected` или `requires_mapping` | "Нужно исправить данные для учета" | передать понятную причину владельцу документа |
| Duplicate | 1С сообщает, что документ уже есть | если контрольные поля совпали, принять как `accepted`; если нет - конфликт | конфликт в review queue | "Документ уже есть в 1С, требуется сверка" | сравнить номер, дату, сумму, контрагента |
| Conflict | внешний id указывает на другой объект или критичные поля не совпали | retry запрещен | `requires_mapping` или `dead_letter` | "Найдено расхождение с 1С" | ручной выбор маппинга или корректировка |
| Partial success | часть строк/документов принята, часть отклонена | повторять только отклоненные элементы с отдельными keys | по rejected items | "Часть данных передана, часть требует исправления" | разобрать item-level ошибки |
| Stale mapping | маппинг устарел, объект архивирован или изменен владельцем | retry запрещен до обновления маппинга | `requires_mapping` | "Нужно обновить сопоставление" | открыть задачу MDM/mapping |
| Unauthorized | токен отозван, нет прав, база 1С отказала | retry запрещен до исправления доступа | scope-level hold | "Нет доступа к обмену с 1С" | проверить токены, права, организацию |
| Schema mismatch | версия формата не совпадает | retry запрещен для scope | scope-level hold | "Формат обмена требует обновления" | согласовать версию схемы и обработку 1С |
| Rate limit / overload | 1С или шлюз ограничивает запросы | retry с увеличенным backoff и лимитом параллелизма | после превышения лимита попыток | "Обмен замедлен, данные будут переданы позже" | настроить расписание и лимиты |
| Manual import file error | файл не читается, кодировка или структура нарушена | retry не нужен, нужен новый файл | `failed` | "Файл обмена не удалось обработать" | проверить формат, кодировку, контрольные поля |
| Bank fact mismatch | банковский факт не сопоставлен с платежом | автоматический retry не применим | review queue | "Платеж из банка требует сверки" | проверить сумму, дату, назначение и контрагента |

## Backoff

Целевая политика для технических ошибок:

| Попытка | Задержка | Примечание |
| --- | --- | --- |
| 1 | сразу | первая отправка |
| 2 | 1 минута | быстрый повтор при сетевом сбое |
| 3 | 5 минут | краткосрочный сбой |
| 4 | 15 минут | стабильная недоступность |
| 5 | 1 час | длинный повтор |
| 6 | 3 часа | последняя автоматическая попытка |

После последней попытки сообщение уходит в `dead_letter` или review queue. Для бизнес-ошибок backoff не используется.

## Idempotency и защита от дублей

Правила:

- каждый документ отправляется с ключом `organization + scope + entity_type + entity_id + business_version`;
- если документ уже принят 1С с тем же ключом, повторная попытка обновляет статус, а не создает новый объект;
- если 1С не поддерживает idempotency key, ProHelper хранит контрольный набор: номер, дата, сумма, контрагент, организация, внешний id;
- duplicate с совпадающими контрольными полями считается безопасным повтором;
- duplicate с отличающимися контрольными полями считается конфликтом;
- при timeout перед повтором нужно попытаться прочитать состояние документа в 1С, если интеграционный формат это поддерживает.

## Журнал попыток

Целевой журнал должен хранить:

| Поле | Назначение |
| --- | --- |
| `run_id` | запуск или batch обмена |
| `message_id` | отдельное сообщение или документ |
| `attempt_id` | конкретная попытка |
| `organization_id` | организация |
| `scope` | контур обмена |
| `direction` | направление |
| `local_entity_type`, `local_entity_id` | локальная сущность |
| `external_id` | внешний id, если есть |
| `idempotency_key` | ключ защиты от дублей |
| `correlation_id` | сквозная связь |
| `status` | queued/sent/accepted/rejected/failed/retrying/requires_mapping/dead_letter |
| `attempt_number` | номер попытки |
| `next_retry_at` | время следующего retry |
| `safe_error_code` | безопасный код ошибки |
| `safe_error_message` | текст для пользователя без секретов |
| `technical_error_ref` | ссылка для поддержки, без вывода пользователю |
| `data_hash` | контрольная сумма состава данных |
| `created_at`, `sent_at`, `finished_at` | временные отметки |

## Безопасные ошибки

Пользовательский текст должен отвечать на вопрос, что делать дальше:

| Код | Текст для пользователя | Что скрывается |
| --- | --- | --- |
| `ONEC_UNAVAILABLE` | "1С временно недоступна. Повторная отправка запланирована." | адрес сервера, stack trace, токены |
| `ONEC_TIMEOUT` | "Не получили подтверждение от 1С. Проверяем статус перед повтором." | low-level transport details |
| `ONEC_VALIDATION_FAILED` | "1С не приняла документ из-за данных. Проверьте указанные поля." | raw response, внутренние constraint names |
| `ONEC_MAPPING_REQUIRED` | "Нужно сопоставить объект с 1С." | внутренние ids без необходимости |
| `ONEC_DUPLICATE_CONFLICT` | "В 1С найден похожий документ с отличающимися данными." | raw payload |
| `ONEC_SCHEMA_MISMATCH` | "Формат обмена отличается от ожидаемого. Требуется обновление настройки." | версию обработчика, если она чувствительна |
| `ONEC_ACCESS_DENIED` | "Нет доступа к обмену с 1С. Обратитесь к администратору." | токены, секреты, права ОС |

## Метрики актуальности

| Метрика | Уровень | Что показывает |
| --- | --- | --- |
| `last_successful_sync_at` | org + scope + direction | когда последний раз обмен был успешным |
| `oldest_pending_message_age` | org + scope | насколько давно висит самый старый документ |
| `retrying_count` | org + scope | сколько сообщений ждут повторной попытки |
| `failed_count` | org + scope | сколько сообщений упало технически |
| `dead_letter_count` | org + scope | сколько сообщений требуют ручного разбора |
| `requires_mapping_count` | org + scope | сколько объектов ждут сопоставления |
| `mapping_coverage_percent` | org + scope | доля объектов с валидным маппингом |
| `accounting_posted_lag` | документный контур | лаг между approval в ProHelper и posted в 1С |
| `bank_fact_lag` | платежи | лаг между банковским фактом и отражением в ProHelper |
| `reconciliation_mismatch_count` | BI/сверка | число активных расхождений |
| `schema_version_mismatch` | integration scope | несовпадение версий формата обмена |

## Пороговые правила

| Условие | Уровень реакции |
| --- | --- |
| Нет успешного обмена по критичному payment scope больше 30 минут | warning |
| Нет успешного обмена по payment scope больше 2 часов | alert |
| `oldest_pending_message_age` больше 1 рабочего дня | alert |
| `dead_letter_count` больше 0 для платежей или актов | alert |
| `requires_mapping_count` растет 3 дня подряд | MDM task |
| `schema_version_mismatch = true` | остановить scope до обновления |
| Payroll package `sent` без accepted/rejected больше 1 рабочего дня | alert workforce/accounting |

## Ручной повтор

Ручной повтор разрешен, если:

- документ не `posted/accounted`;
- причина ошибки техническая или исправленная;
- idempotency key сохранен;
- пользователь имеет право на scope;
- журнал содержит предыдущую попытку и причину.

Ручной повтор запрещен, если:

- есть unresolved conflict;
- требуется mapping;
- 1С уже провела документ с отличающимися контрольными полями;
- документ отменен или заменен в ProHelper;
- source hash payroll-пакета устарел.

## Актуальность данных в UI

Пользовательские экраны должны показывать:

- дату последнего успешного обмена;
- статус текущей передачи;
- понятную причину отказа;
- срок следующего retry, если он запланирован;
- признак, что данные в 1С могут быть неактуальны;
- отдельный статус "отражено в учете" только после подтверждения 1С.

Не следует показывать пользователю низкоуровневые термины вроде stack trace, SQL, raw payload, token hash, internal exception.

## Открытые вопросы

- Где хранить dead-letter: в общей таблице exchange messages или как отдельную очередь поддержки.
- Сколько автоматических retry допустимо для каждого scope: платежи, акты, склад, payroll-source.
- Нужно ли блокировать отправку новых документов по scope, если обнаружен schema mismatch.
- Кто отвечает за ручной retry и manual resolution в каждой роли: бухгалтер, администратор, поддержка.
- Нужна ли отдельная страница мониторинга обмена для организации и отдельная системная панель для поддержки ProHelper.
