# Матрица источников истины ProHelper / 1С / внешние системы

Дата: 2026-06-07
Задача: PHERP-57
Статус документа: аналитика для проверки

## Принцип

ProHelper должен быть источником истины по операционному строительному контуру: проект, график, смета, закупка, склад на объекте, платежный workflow, документы стройки, роли, полевые факты и управленческая аналитика.

1С должна быть источником истины по бухгалтерскому и налоговому учету: юридические реквизиты, бухгалтерские документы, проводки, план счетов, регламентированное закрытие, налоговая база, отчетность и юридически значимая зарплата.

Внешние системы остаются источником истины там, где они фактически исполняют или юридически подтверждают операцию: банк/клиент-банк, ЭДО/КЭП, нормативные базы, адресные классификаторы, зарплатная система, CRM при ее наличии.

## Где сейчас живут данные

| Группа данных | Сейчас |
| --- | --- |
| Операционная стройка | В ProHelper backend и клиентах: projects, schedule, daily plans, quality, safety, handover, executive docs, production labor. |
| Управленческие финансы | В ProHelper Payments/Finance/Reports: заявки, schedules, документы, транзакции, привязки к договорам/проектам. |
| Бухгалтерские факты | Полноценного локального accounting-core нет. Есть поля `external_code`, `accounting_data`, one-c mappings/runs и сервисы обмена. |
| Регламентированная первичка | Частично готовится в ProHelper как операционный документ; юридическая сила должна приходить из 1С/ЭДО/внешнего контура. |
| Справочники | MDM/каталоги ProHelper плюс потенциальная синхронизация с 1С и внешними классификаторами. |
| Payroll | ProHelper хранит явку, выработку, source rows/statements/export packages; юридический расчет зарплаты должен оставаться внешним. |

## Запрещенные зоны без отдельного решения

Следующие зоны нельзя строить в ProHelper как часть текущего ERP-roadmap без отдельного ADR, бюджета, требований бухгалтерии/юристов и интеграционного проекта:

- полноценный бухучет;
- налоговый учет;
- регламентированная отчетность;
- юридически значимый payroll;
- юридически значимый ЭДО/КЭП;
- самостоятельное банковское исполнение платежей.

## Минимальные требования к промышленной синхронизации

1. Единый `external_id`/mapping per organization и, при необходимости, per 1С base.
2. Очереди, retry, idempotency keys и запрет повторного создания одного и того же документа.
3. Отдельные статусы: operational status, legal signature status, bank execution status, accounting posted status.
4. Field-level ownership для контрагентов, договоров, материалов, статей бюджета, сотрудников.
5. Конфликтные изменения только через review queue, а не silent overwrite.
6. Журнал обмена с безопасными ошибками, без раскрытия секретов.
7. Отчеты сверки: ProHelper vs 1С vs банк/ЭДО.
