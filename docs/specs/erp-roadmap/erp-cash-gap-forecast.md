# Прогноз кассового разрыва

Задача: PHERP-86, родительская задача PHERP-24.

Дата проектирования: 2026-06-08.

Статус: спецификация и чистое расчетное ядро backend. API endpoint и агрегаторы источников данных должны подключаться отдельным внедренческим шагом без изменения бухгалтерского контура.

## Цель

Cash gap forecast в ProHelper должен показывать управленческий прогноз БДДС по дням: когда и почему ожидается отрицательный денежный остаток, какие поступления, выплаты, резервы и просрочки формируют риск, и какие сценарные допущения повлияли на расчет.

Это не бухгалтерский учет, не налоговый учет, не регламентированная отчетность и не юридически значимый payroll. 1С остается source of truth для бухгалтерии и налогов. ProHelper остается source of truth для строительного ERP-контура: проектов, бюджетов, лимитов, платежных заявок, платежного календаря, управленческого forecast и объяснения рисков.

Связанные решения:

- PHERP-80: модель БДР/БДДС, статей бюджета, ЦФО и границы с 1С;
- PHERP-81: backend/admin API бюджетирования и импорт бюджетов;
- PHERP-82: расчет доступного лимита платежей;
- PHERP-83: интеграция лимитов в платежный контур и runtime-резервы.

## Границы решения

В рамках PHERP-86 реализуется чистое расчетное ядро:

- `App\BusinessModules\Features\Budgeting\Services\CashGapForecastService`;
- DTO входных фильтров, контекста, событий, дневных строк и результата в `App\BusinessModules\Features\Budgeting\DTOs`;
- unit tests без обращения к БД.

Ядро не читает базу данных и не принимает решений о правах. Оно получает уже нормализованные денежные события и фильтры, рассчитывает дневной прогноз и возвращает бизнес-понятный контракт. Это позволяет отдельно тестировать алгоритм и позже подключить production-sized агрегаторы поверх `PaymentTransaction`, `PaymentDocument`, `PaymentSchedule`, `BudgetLimitReservation`, активной БДДС-версии бюджета и ручных допущений.

## Источники данных

| Источник | Роль в прогнозе | Правило использования |
| --- | --- | --- |
| Текущие остатки денежных средств | `opening_balance` на начало горизонта | берутся из управленческого источника остатков или сверенной банковской позиции; не моделируются бухгалтерскими проводками |
| `PaymentTransaction` со статусом `completed` | фактические inflows/outflows | дата берется из `transaction_date` или согласованной value date; это факт БДДС |
| Сверенные банковские выписки | фактические inflows/outflows | после сверки могут дополнять или подтверждать `PaymentTransaction`; дубль одной операции запрещен |
| `PaymentDocument` | плановые обязательства и поступления | approved/scheduled/partially_paid дают будущие потоки по `remaining_amount` |
| `PaymentSchedule` | график оплат | используется как более точная дата, если документ разбит на платежи |
| `BudgetLimitReservation` | `reserved_outflows` | активные резервы уменьшают прогнозный остаток до оплаты или освобождения |
| Активная БДДС-версия бюджета | плановые/forecast потоки | используется как план и контрольный baseline, а не как факт денег |
| Просроченные входящие платежи | `overdue_inflows` | показываются отдельным блоком риска и не увеличивают остаток без переноса или подтверждения даты |
| Просроченные исходящие платежи | `overdue_outflows` | показываются отдельным блоком риска и учитываются как немедленное обязательство на старте горизонта |
| Ручные допущения | сценарные корректировки | должны иметь автора, причину, срок действия и audit trail |

`one_c_base_id` и `integration_profile_id` могут использоваться только как идентификаторы интеграционного контура для сверки и mapping. Они не должны превращаться в модель проводок, налоговых регистров или бухгалтерских документов 1С.

## Статусы платежей

| Сущность | Включается | Роль |
| --- | --- | --- |
| `PaymentTransaction.completed` | да | факт поступления или выплаты |
| `PaymentDocument.approved` | да | утвержденное обязательство или ожидаемое поступление |
| `PaymentDocument.scheduled` | да | платеж в календаре |
| `PaymentDocument.partially_paid` | да | только неоплаченный остаток через `remaining_amount` |
| `PaymentDocument.submitted` | опционально | отдельный слой риска или pending commitment, если финансовая политика включает заявки до финального согласования |
| `PaymentDocument.pending_approval` | опционально | отдельный слой риска или pending commitment |
| `PaymentSchedule.pending` | да | плановая дата и сумма части платежа |
| `BudgetLimitReservation.reserved` | да | резерв исходящего потока |

Исключаются из денежного прогноза:

- `draft`: черновик не является обязательством;
- `rejected`: отклонено;
- `cancelled`: отменено;
- `PaymentTransaction.failed`, `cancelled`, `refunded`: не является положительным фактом БДДС;
- `PaymentDocument.paid` как плановый поток: факт должен прийти из завершенных транзакций, а не из денормализованного состояния документа.

## Приоритет дат

Для каждого события выбирается одна расчетная дата:

| Приоритет | Дата | Когда применяется |
| --- | --- | --- |
| 1 | фактическая дата оплаты или поступления | `PaymentTransaction.completed`, сверенная банковская операция |
| 2 | дата `PaymentSchedule.due_date` | документ имеет график платежей |
| 3 | `PaymentDocument.scheduled_at` | платеж поставлен в календарь |
| 4 | `PaymentDocument.due_date` | есть срок оплаты или поступления |
| 5 | `PaymentDocument.document_date` | fallback только для плановых документов без срока |
| 6 | дата ручного допущения | сценарная корректировка |

Просроченные входящие и исходящие платежи не смешиваются с обычными плановыми потоками. Просроченный входящий платеж остается в `overdue_inflows` до ручного переноса даты или подтверждения новой вероятности. Просроченный исходящий платеж попадает в `overdue_outflows` на дату старта прогноза, потому что это уже наступившее обязательство.

Если платеж перенесен, в событии сохраняются расчетная `date` и `original_date`; объяснение риска должно показывать, что дата изменилась.

## Daily Cash Balance

Для каждого дня горизонта строится строка:

```text
opening_balance =
  previous_day.closing_balance
  или opening_balance горизонта для первого дня

closing_balance =
  opening_balance
  + inflows
  - outflows
  - reserved_outflows
  - overdue_outflows

cash_gap = max(0, -closing_balance)
```

`overdue_inflows` не прибавляются к остатку автоматически: это риск непоступления, а не деньги. Они показываются в объяснении и могут изменить сценарий после ручного переноса даты.

Daily contract:

```json
{
  "date": "2026-06-18",
  "opening_balance": 1500000.0,
  "inflows": 300000.0,
  "outflows": 1200000.0,
  "reserved_outflows": 400000.0,
  "overdue_inflows": 600000.0,
  "overdue_outflows": 500000.0,
  "closing_balance": -300000.0,
  "cash_gap": 300000.0,
  "risk_level": "critical",
  "explanation": {},
  "drivers": []
}
```

## Сценарии

Алгоритм поддерживает три сценария:

| Сценарий | Поступления | Выплаты и резервы | Назначение |
| --- | --- | --- | --- |
| `optimistic` | вероятность поступления повышается на утвержденный lift, дата может быть сдвинута раньше | обязательства считаются в полной сумме | верхняя граница при благоприятной собираемости |
| `base` | сумма планового поступления умножается на вероятность | обязательства считаются в полной сумме | основной управленческий прогноз |
| `stress` | вероятность дополнительно умножается на stress-factor, дата поступления переносится позже | обязательства считаются в полной сумме | проверка устойчивости при задержках денег |

В текущем ядре значения по умолчанию:

- `stress_inflow_delay_days = 7`;
- `stress_inflow_probability_factor = 0.75`;
- `optimistic_inflow_probability_lift = 0.1`;
- `optimistic_inflow_advance_days = 0`.

Фактические операции не сдвигаются и не умножаются на вероятность. Просроченные входящие платежи остаются отдельным блоком риска во всех сценариях, пока не задана новая дата.

## Вероятности и переносы дат

Вероятность применяется только к будущим входящим потокам. Исходящие платежи, резервы и просроченные обязательства считаются по полной сумме, потому что для cash gap важна худшая сторона обязательств.

Рекомендованные источники вероятности:

- статус договора и актов;
- история оплат контрагента;
- подтверждение заказчика;
- просрочка и количество переносов;
- ручное финансовое допущение с причиной.

Перенос даты должен хранить:

- исходную дату;
- новую расчетную дату;
- автора и причину;
- срок действия допущения;
- ссылку на платеж, договор, проект или контрагента.

## Объяснение риска

На любой день с отрицательным остатком пользователь должен видеть:

- начальный остаток дня;
- какие inflows ожидались и с какой вероятностью;
- какие outflows, scheduled payments, approved payments и reserves уменьшили остаток;
- какие overdue inflows не были засчитаны как деньги;
- какие overdue outflows были учтены как срочное обязательство;
- какие события имеют `original_date`, если платеж перенесен;
- минимальный остаток, сумму разрыва и дату первого разрыва.

В API это выражается блоками:

- `explanation`: summary, first gap date, negative days, max gap amount, min closing balance, overdue totals;
- `drivers`: список источников с `type`, `direction`, `date`, `original_date`, `amount`, `balance_impact`, `probability`, `source`.

Пользовательские тексты не должны содержать `payload`, `dto`, `exception`, `sql`, `constraint`, stack trace, raw payload или секреты. PHP-сообщения возвращаются через `trans_message(...)`.

## Фильтры

Поддерживаемые фильтры:

- `organization_id`;
- `project_id`;
- `counterparty_id`;
- `budget_article_id`;
- `responsibility_center_id`;
- `period_from` / `period_to`;
- `currency`.

Фильтр по организации является границей безопасности. Событие с чужой `organization_id` не должно попадать в расчет даже при совпадении проекта, статьи или контрагента. Остальные фильтры применяются как уточнение управленческого среза.

## Production-sized агрегирование

Для реального endpoint нельзя загружать все документы и фильтровать их в PHP. Нужно собирать источники пакетно:

- фактические транзакции группировать по дате, направлению, валюте и аналитикам;
- платежные документы группировать по расчетной дате и статусу;
- активные резервы группировать по `period_month`, валюте и аналитикам;
- schedules агрегировать до уровня дня до соединения с документами;
- drill-down источники возвращать ограниченно и постранично, а не вкладывать все документы в дневную строку;
- индексы должны покрывать `organization_id`, дату, статус, валюту, проект, статью бюджета и ЦФО.

Расчетное ядро должно получать уже нормализованные `CashGapForecastItem[]`. Это отделяет алгоритм от ORM, предотвращает N+1 и упрощает тестирование.

## Edge Cases

| Сценарий | Ожидаемое поведение |
| --- | --- |
| Частично оплаченный платеж | в прогноз идет только `remaining_amount`, факт уже должен быть в `PaymentTransaction.completed` |
| Просроченный входящий платеж | показывается в `overdue_inflows`, не увеличивает остаток |
| Просроченный исходящий платеж | показывается в `overdue_outflows`, уменьшает остаток на старте горизонта |
| Платеж перенесен | расчет идет по новой дате, `original_date` сохраняется в driver |
| Несколько платежей в один день | суммы агрегируются в одной дневной строке |
| День с отрицательным closing balance | `cash_gap.has_gap = true`, риск `critical` |
| Разные сценарии | stress может дать более позднюю дату поступления и другую сумму риска |
| Нет данных | возвращается валидный прогноз с нулевыми потоками и дневными строками |
| Чужая организация | событие не попадает в расчет |

## Ограничения

Не входит в PHERP-86:

- бухгалтерские проводки, план счетов, налоговые регистры;
- регламентированная отчетность;
- юридически значимый payroll;
- автоматическое чтение production-остатков;
- миграции новых таблиц;
- UI и браузерный сценарий;
- endpoint с RBAC, потому что сначала нужно утвердить источники opening balance и правила drill-down.
