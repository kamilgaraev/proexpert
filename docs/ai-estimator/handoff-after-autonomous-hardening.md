# AI-сметчик: handoff после автономного hardening

Дата: 2026-06-28

## Что уже сделано автономно

1. Старый модуль `AIEstimates` удален из backend и admin UI как нецелевой legacy-код.
2. Основной поток переведен на модуль `EstimateGeneration`: документация и чертежи загружаются в сессию, далее система распознает источники, извлекает факты, элементы чертежей, объемы и предполагаемые виды работ.
3. Добавлены таблицы и модели для результатов понимания документов: страницы OCR, факты, элементы чертежей, объемы, выводы по составу работ.
4. Добавлен строгий backend-контракт `estimator_readiness`. Он показывает:
   - можно ли запускать генерацию;
   - можно ли сохранять смету;
   - какие блокеры мешают;
   - какие предупреждения есть;
   - какие метрики разобраны по документам, чертежам, нормам и ценам.
5. `apply` теперь блокируется на backend, если документы не готовы, есть документы на проверку, нет рассчитанных позиций, нормы требуют проверки или качество черновика заблокировано.
6. Admin workspace показывает проверку документов и чертежей: страницы, факты, элементы чертежей, извлеченные объемы, выводы по работам.
7. Admin workspace показывает readiness-панель: следующий шаг, статус, блокеры, предупреждения, документы, объемы с чертежей, факты/элементы и позиции с ценой.
8. Кнопка сохранения сметы зависит от backend `can_apply`, а не только от локальной логики интерфейса.
9. Добавлены unit/Vitest-тесты на readiness, resource-контракт и UI helper-логику.
10. Добавлен план автономного hardening: `docs/superpowers/plans/2026-06-28-ai-estimator-autonomous-hardening.md`.

## Что это означает для продукта

Текущая архитектура больше не говорит пользователю “смета готова”, если система не разобрала входные документы или не может безопасно привязать позиции к нормативной базе. Это не полноценная замена инженера-сметчика на реальных проектах до подключения боевых данных и валидации, но теперь поток честно отделяет:

- распознавание документов и чертежей;
- извлечение объемов;
- подбор норм ФСНБ/ФЕР/ГЭСН и ресурсов;
- расчет по региональным ценам;
- блокировку опасного применения;
- проверку спорных мест человеком.

## Что должна сделать команда

1. Применить миграции в нужном окружении после code review.
2. Проверить, что queue workers/Horizon настроены для очередей OCR, разбора документов и генерации смет. Без воркеров большие загрузки будут зависать или выполняться нестабильно.
3. Проверить S3-хранилище: все загруженные документы должны храниться в папках организации, локальное хранение для этого потока недопустимо.
4. Настроить OCR-провайдера и лимиты: таймауты, retry, лимиты размера файла, лимиты страниц, обработку частичных ошибок.
5. Импортировать и проверить нормативную базу: ФСНБ, ФСБЦ/ресурсы, ФЕР/ГЭСН, региональные цены, кварталы, зоны и версии.
6. Собрать эталонный набор: реальные PDF/DWG/сканы/ведомости + сметы из Гранд-Сметы для сравнения.
7. Для каждого эталонного проекта зафиксировать допустимые отклонения по объемам, составу работ, нормам и стоимости.
8. Подключить реальный vision/CAD-анализ чертежей. Текущий rule-based слой помогает извлекать текстовые признаки и простые объемы, но не заменяет полноценное понимание планов, разрезов, узлов, спецификаций и DWG.
9. Утвердить правила ручной проверки: какие блокеры запрещают сохранение, какие предупреждения допускаются, кто подтверждает спорные нормы.
10. Настроить production-наблюдаемость: длина очередей, время OCR, время генерации, процент документов на проверку, процент позиций без нормы, ошибки провайдера.

## Ручная приемка

Проверить минимум 5 реальных кейсов:

1. Только описание без документов: система должна попросить загрузить документацию и не обещать точную смету.
2. PDF с текстовой проектной документацией: должны появиться факты, страницы и источники.
3. Чертежи/сканы низкого качества: сессия должна уйти в проверку, а сохранение сметы должно быть заблокировано.
4. Документы с извлеченными объемами: readiness должен показать объемы с чертежей и разрешить генерацию.
5. Черновик с позициями без безопасной нормы: сохранение должно быть заблокировано до проверки норм.

## Ограничения, которые нельзя скрывать

- Без реального корпуса чертежей и эталонных смет нельзя утверждать, что система уже заменяет сметчика.
- Без полноценного vision/CAD-пайплайна система не будет надежно понимать сложные чертежи.
- Без актуальных нормативных и региональных цен результат не может быть близок к Гранд-Смете.
- Без фоновых очередей и лимитов несколько больших генераций могут перегрузить backend.
- Без доменной валидации сметчиком нельзя безопасно снижать пороги проверки.

## Следующий технический этап

1. Подключить production-grade OCR/vision/CAD provider contract для чертежей.
2. Сделать benchmark runner: входные документы -> AI-смета -> сравнение с эталоном из Гранд-Сметы.
3. Добавить метрики точности по документам, объемам, нормам и стоимости.
4. Перевести спорные нормы в review workflow с подтверждением/отклонением и обучающей обратной связью.
5. Добавить нагрузочный сценарий очередей на пакетные загрузки, но выполнять его только в согласованном окружении.
