# Обучающие наборы AI-сметчика в суперадминке

## Что реализовано

В backend-суперадминке появился раздел `AI-сметчик -> Обучающие сметы`.

Раздел принимает эталонный набор:

- эталонную смету из Гранд-Сметы, МОСТ или ручного источника;
- проектную документацию;
- чертежи;
- сканы;
- ведомости;
- регион и период цен;
- оценку доверия к эталонной смете.

После загрузки набор можно отправить в очередь обработки. Обработка:

- сохраняет файлы в S3 в папку организации;
- определяет формат эталонной сметы существующим runtime-import pipeline;
- разбирает строки сметы;
- отделяет работы от разделов, итогов и ресурсных дочерних строк;
- нормализует коды норм;
- проверяет наличие нормы в нормативной базе;
- проверяет совместимость единицы измерения;
- создает `EstimateGenerationLearningExample`;
- связывает созданный обучающий пример с исходной строкой эталона.

## Правильная структура набора

Один набор должен описывать один реальный строительный кейс:

- одна эталонная смета, которую можно считать правильной;
- все документы, по которым эта смета была составлена;
- чертежи и сканы, если по ним брались объемы;
- ведомости объемов, если они есть;
- понятные регион и период цен;
- источник эталона: Гранд-Смета, МОСТ или ручная проверка.

Не стоит смешивать в одном наборе разные объекты, разные регионы или разные версии смет. Для этого нужно создавать отдельные наборы.

## Что считается хорошим эталоном

Эталон можно использовать для обучения, если:

- смета составлена сметчиком или выгружена из Гранд-Сметы;
- в позициях есть коды норм ФСНБ/ФЕР/ТЕР;
- единицы измерения совпадают с нормативной базой;
- объемы связаны с документацией или ведомостями;
- понятны регион и период цен;
- нет ручных строк без нормативного основания, либо они явно отделены.

## Что команда должна подготовить

Для реального повышения качества AI-сметчика нужна библиотека эталонов:

- 30-50 малых объектов для первой калибровки;
- 100-300 объектов для устойчивого подбора норм;
- реальные пары `документы + чертежи + эталонная смета`;
- отдельные наборы по типам работ: бетон, кладка, отделка, инженерка, кровля, демонтаж;
- отдельные наборы по регионам и периодам цен;
- экспорт из Гранд-Сметы в машиночитаемом формате, а не только PDF.

## Что эта загрузка не делает сама

Текущий этап не дообучает внешнюю LLM напрямую. Он создает проверяемую базу обучающих примеров внутри системы. Эти примеры уже можно использовать как RAG/контекст для подбора норм и как датасет для будущего fine-tuning/evaluation pipeline.

Для полноценного обучения модели дальше нужны:

- offline evaluation runner: сравнение AI-сметы с эталоном по нормам, объемам, секциям и стоимости;
- датасет-версии, чтобы понимать, на каких примерах проверялась модель;
- pipeline для генерации train/validation/test splits;
- vision/CAD pipeline для извлечения объемов из чертежей;
- отдельная модель качества, которая будет показывать, где смета требует проверки сметчиком.

## Как проверять результат

После обработки набора смотри:

- `Строк разобрано`;
- `Строк принято`;
- `Строк пропущено`;
- `Примеры обучения`;
- статус качества.

Если принято мало строк, обычно причина одна из трех:

- в эталонной смете нет кодов норм;
- код нормы не найден в базе нормативов;
- единица измерения в смете не совпадает с единицей нормы.

Такой набор не нужно удалять сразу: его можно использовать для диагностики импорта, но для обучения он слабый.
